# Editora

A keyboard-driven, cross-platform programmer's text editor. JDK 25 + JavaFX 25 +
Maven, modular (JPMS, module `com.editora`).

## Commands

- Run the app: `mvn javafx:run`
- Run tests: `mvn test`
- Build app image / native installer: `mvn -Pdist package`
  - Produces `target/dist/Editora.app` (macOS); OS profiles auto-select DMG/MSI/DEB.
  - Quick unpackaged bundle: `mvn -Pdist -DskipTests -Djpackage.type=APP_IMAGE package`
- Build a runnable fat jar: `mvn -Pfatjar package` ⇒ `target/Editora-<version>.jar`, run with
  `java -jar`. It bundles JavaFX (classes + natives) for **the build host's platform only** and runs
  from the classpath via the non-`Application` `com.editora.Launcher` main class. A single
  all-platforms jar is impossible (JavaFX's macOS/Linux x64 and arm64 natives share filenames and
  collide), so the release CI builds one fat jar per runner.
- Cut a release: bump `<version>` in `pom.xml`, push a `vX.Y.Z` tag (`-rcN` ⇒ pre-release).

Run Maven from the project root (`/Users/adriandeleon/src/adl/Editora-V2`).

## Worktrees — one per task (important)

This repo is worked on by multiple Claude Code sessions. **Each task gets its own
`git worktree`** so sessions don't share a single working tree (which previously caused commits to
land on the wrong branch when a parallel session ran `git checkout` in the shared checkout).

- **Start a task:** `scripts/worktree.sh new <branch>` (e.g. `feat/foo`, base defaults to
  `origin/master`). It creates an isolated checkout at `../Editora-V2-worktrees/<slug>` on a new
  branch; `cd` into it and do the work there. `list` shows all worktrees; `rm <branch>` removes the
  worktree + branch after merging; `prune` cleans stale metadata.
- **Never `git checkout` a different branch in the main checkout** (`/Users/adriandeleon/src/adl/Editora-V2`)
  while other sessions may be active — switch by creating/entering a worktree instead. Keep the main
  checkout on `master`.
- **Subagents:** spawn them with the Agent tool's `isolation: "worktree"` so background work can't
  disturb the main checkout.
- **A session/terminal does not "move into" a worktree by creating one.** A session is anchored to
  the directory it launched in; a worktree is just a separate folder. To actually *work inside* one
  interactively, open a **new terminal / Claude Code session** in that directory
  (`cd ../Editora-V2-worktrees/<slug>`). A long-running session pinned to the main checkout (e.g. the
  one whose terminal sits on `master`) instead operates on a worktree **by path** — absolute paths for
  edits, `cd <worktree> && ./mvnw …` for builds — and stays on `master`; commit/merge from there but
  **never `git checkout` another branch in the main checkout**. (Claude's Bash tool also resets its
  working dir to the project root after each command, so it can only reach a worktree by path, not by
  persistently `cd`-ing in.)
- Worktrees live in the sibling `../Editora-V2-worktrees/` dir (outside the repo), so nothing needs
  to be gitignored. They share the same `.git`, so all branches/commits are visible everywhere.

## Release pipeline

`.github/workflows/release.yml` runs on a `v*` tag (or manual dispatch for a dry run): a 5-way
matrix (linux x64/arm64, macOS x64/arm64, windows x64 — **Windows arm64 is omitted: a hosted
runner now exists (`windows-11-arm`, GA Jan 2026), but OpenJFX 25 publishes no `win-aarch64` native
jar on Maven Central (see [JDK-8314064]), so a native ARM64 build can't link — Windows-on-ARM users
run the x64 installer under x64 emulation. Revisit when JavaFX ships win-aarch64 natives.** macOS x64
uses the `macos-15-intel` runner — the last Intel x86_64 image (good through ~Aug 2027) — since the
old `macos-13` Intel runner was retired Dec 2025;
each on its own GitHub-hosted runner) builds the native
installer via the existing `-Pdist` profile — there is **no cross-building** (jpackage + JavaFX are
host-specific), so each runner builds for itself. Each runner also builds a per-platform runnable
fat jar via `-Pfatjar` (`Editora-<version>-<target>.jar`). Installers are renamed to
`Editora-<version>-<target>.<ext>` per target — one consistent `Editora-<version>-<target>` prefix
across all artifacts (jpackage's DMG/MSI names omit the version + arch; the
version comes from a `Resolve version` step — the tag minus `v`, else the pom version) and uploaded as
artifacts alongside the fat jar; a final job
hands them to **JReleaser** (`jreleaser.yml`, via `jreleaser/release-action`) which creates the
GitHub release with all installers + fat jars + `checksums.txt` + a changelog. JReleaser only *orchestrates the release* — it does not
build (the `dist` profile is reused as-is) and there is **no `pom.xml`/Maven change**, so the normal
build is unaffected. Installers are currently **unsigned** (signing/notarization is a follow-up).
Uses the BellSoft **Liberica** JDK 25 in CI for full arch coverage (incl. linux aarch64).

## Architecture

Single module `com.editora` (see `src/main/java/module-info.java`). `App` is the
JavaFX entry point: it loads config, builds the command + keymap system, installs
the key dispatcher on the scene, applies the theme, and loads `ui/main.fxml`. It
wraps the FXML root in a `StackPane` so a floating overlay can sit on top — used by
`MainController.installZenOverlay` for the **Zen-mode exit button** (a small floating "Z",
`Icons.zen()`, styled `.zen-exit` like the Markdown preview controls, shown top-right only while
`WorkspaceState.isZenMode()` — toggled in `applyChromeVisibility` via `updateZenButton` — and
clicking it runs `setZenMode(false)`). `updateZenButton` also drops the Z below the Markdown preview
controls (a larger top margin) when the active buffer is Markdown, since those also float top-right; it
re-runs on tab switches so the offset tracks the active file. `installZenOverlay` also creates a second
floating button, the **toolbar-restore "Tool" button** (`Icons.tools()`, styled `.toolbar-restore` like
the Zen "Z", also `Pos.TOP_RIGHT` — it never coexists with the "Z", which only shows in Zen mode), shown by `updateToolbarRestoreButton` (called from
`applyChromeVisibility`) only when the toolbar is hidden and not in Zen (`!isShowToolbar() && !isZenMode()`);
clicking it runs `toggleToolbar()` (which also `settingsWindow.syncToolbarCheck()`s), so a user who hides
the toolbar (Settings checkbox or the `view.toggleToolbar` palette command) can always bring it back. The
**toolbar's state-aware edit icons** (Save/Undo/Redo/Cut/Copy/Paste) are kept enabled/disabled by
`refreshEditState()` (save←dirty, undo/redo←`area.isUndoAvailable/isRedoAvailable`, cut←selection+editable,
copy←selection — all disabled on a non-buffer/Welcome tab) and `refreshPasteState()` (paste←editable +
`Clipboard.hasString()`, kept off the per-keystroke path: run on tab switch, window focus-regain, and after
cut/copy). `refreshEditState` runs on tab switch and the active buffer's edit/selection/dirty pulses;
disabling is cosmetic only — the `C-s`/`C-z`/… keymap commands still work. The toolbar's **Find in Files**
icon (`Icons.findInFiles()`, `onFindInFiles → openSearchInFiles`) sits beside the in-file Find icon.

- `command/` — the keyboard-driven core.
  - `Command` / `CommandRegistry`: every action is a registered `Command` (id + title + runnable). The registry feeds both keybindings and the palette.
  - `KeymapManager`: maps chord sequences (e.g. `C-x C-s`) to command ids; loads `resources/com/editora/keymaps/emacs.json`, then overlays user overrides from config.
  - `KeyDispatcher`: a scene-level `KEY_PRESSED` filter that builds chord tokens and resolves them, holding a pending-prefix buffer for multi-key Emacs chords. Lone unbound keys are *not* consumed, so normal typing works.
- `editor/` — `EditorBuffer` wraps a RichTextFX `CodeArea` (line numbers + fold chevrons via `FoldManager`, debounced highlighting). **Bookmarks**: a per-buffer `BookmarkManager` (analogous to `FoldManager`) holds line→`Bookmark` (record `line/note/lineText` in `config`), tracks lines through edits via `plainTextChanges` so bookmarks **follow their content** with forward/"sticky" gravity — inserting at a bookmarked line's start pushes it down (pure, column-aware `BookmarkManager.shift(...)` is unit-tested). Because `shift` only runs for edits made *inside* the app, on open each bookmark is **re-anchored to its saved `lineText`** (pure, unit-tested `BookmarkManager.reanchor(...)`: keeps an exact match in place, else moves it to the *nearest* line whose stripped text matches — downward-first on ties — bounded by `MAX_REANCHOR_SCAN`; an empty/vanished `lineText` stays at its clamped stored line), so bookmarks survive **external** edits to the file. `restore(...)` returns whether anything moved and `MainController.restoreBookmarks` then calls `persistBookmarks` to self-heal the saved indices once (later opens hit the exact-match fast path, no save). The gutter draws a `.bookmark-marker` glyph (wired through `FoldManager.setBookmarkHooks`, refreshed per-line via `recreateParagraphGraphic`). Bookmarks live in their own `bookmarks.json` (`config/BookmarkStore`) — kept out of `WorkspaceState` so they're all in one easy-to-find file — but **scoped per project**: the store is `Map<projectKey, Map<path,List<Bookmark>>>` (`byProject`; `projectKey` = `""` for the no-project/global session, else the project id). `ConfigManager.getBookmarks()` returns the **active** project's bucket, chosen by `currentBookmarkKey()` (derived from the active `workspaceStateFile`: `workspace-state.json` → `""`, `projects/<id>.json` → `<id>`), so switching a project (`setWorkspaceStateFile`/`useDefaultWorkspaceStateFile`) automatically swaps which bookmarks are visible — `MainController.activateSession` already calls `bookmarksPanel.refresh()`. `saveBookmarks()` writes the whole `bookmarks.json`; `MainController.persistBookmarks`/`restoreBookmarks` call `saveBookmarks()` — *not* `save()`. On first run, `ConfigManager.loadBookmarks()` does a one-time migration: it pulls bookmarks previously embedded in `workspace-state.json` (→ the `""` bucket) and each `projects/<id>.json` (→ that project's bucket), strips the legacy `bookmarks` node from those files, and writes `bookmarks.json` (created even when empty) so the migration runs once. Deleting a project also drops its bookmarks (`MainController.deleteProject` → `ConfigManager.deleteBookmarksForProject(id)`). The `BookmarksPanel` reads the active bucket (so it includes closed files) and routes edits back through the controller (open buffer's manager, else the map). **Order is user-controlled, not sorted**: the panel renders the bucket's map order (files) and each file's list order (bookmarks) verbatim, and the `M-g b` jump picker (`allBookmarkEntries`) reads the same order — so the two always agree. Reorder a bookmark within its file or a whole file group via **Alt+Up/Down**, a right-click **Move Up/Down**, or **drag-and-drop** (bookmark onto a sibling, file header onto another; disabled while a filter is active), routed through `Actions.moveBookmark`/`moveFile` → `MainController.moveBookmark`/`moveBookmarkFile` (mutate the bucket + `saveBookmarks`; the panel reselects the moved row after refresh). The drag uses the **same visuals as the tab strip**: a translucent `setDragView` snapshot follows the cursor, the dragged `.tree-cell` dims (`.bookmark-dragging`), and an accent insertion line shows on the target row's top/bottom edge (`.bookmark-drop-above`/`.bookmark-drop-below`, mirroring `.tab-drop-before/after`); `BookmarksPanel.insertIndex` (pure/unit-tested) maps the drop side to the post-removal insert index. Because an open buffer's `BookmarkManager.snapshot()` is line-ordered, `persistBookmarks` runs the pure, unit-tested `BookmarkStore.mergePreservingOrder(previous, snapshot)` to keep the custom order across edits (matching bookmarks by line, else by captured `lineText` so the order follows content through line shifts). Commands: `bookmarks.toggle` (`C-c m`), `bookmarks.next`/`previous` (`C-c ]`/`C-c [`, within-file), `bookmarks.jump` (`M-g b`, cross-file picker), `editNote`/`clearFile`. **Markdown preview**: an IntelliJ-style per-buffer 3-mode view (`EditorBuffer.MarkdownViewMode` EDITOR/SPLIT/PREVIEW) driven by a floating `MarkdownViewToggle` overlaid top-right of the editor for Markdown files; `MarkdownRenderer` parses CommonMark+GFM (commonmark-java, all proper JPMS modules — `requires org.commonmark.*`, no moditect) **off-thread** then builds native JavaFX nodes **on the FX thread** under a stale-gen guard (the `highlightExecutor` idiom), debounced via `multiPlainChanges().successionEnds(250ms)`, only while SPLIT/PREVIEW (≥5 MB files render once). The mode is persisted per file in `WorkspaceState.markdownViewModes` (`MainController.persistMarkdownMode`/`restoreMarkdownMode`), mirroring folds. Preview colors use AtlantaFX semantic CSS (`.markdown-preview`/`.md-*` in `app.css`) so they track the theme without per-theme overrides. **Independent light/dark preview theme**: the preview can be themed separately from the app/editor theme via a floating sun/moon button added to the `−`/`+` zoom control (top-left). Persisted globally in `Settings.markdownPreviewTheme` (`""` = follow the app theme until first toggled, then `"light"`/`"dark"`; schema bumped 2→3, additive identity migration). `MainController.toggleMarkdownPreviewTheme` (also the palette `view.toggleMarkdownPreviewTheme`) flips the effective theme — seeded from app brightness via `appThemeDark()` (`Themes.backgroundFor`) — saves, and `applyMarkdownPreviewTheme()` pushes it to every open buffer. `EditorBuffer.applyPreviewTheme(mode, appDark)` adds `md-light`/`md-dark` to the preview `ScrollPane`, which **redefine the looked-up `-color-*` vars for the whole preview subtree** so the existing `.md-*` rules re-resolve to a fixed GitHub-style palette (no rule duplication), and refreshes the toggle glyph (sun while dark → switch to light, moon while light → switch to dark). The toggle is injected via `setPreviewThemeToggle` (like the snippet/completion providers — `editor` can't depend on `ui`). No Settings-window entry. For readability the content is a **centered, width-capped column** (`MAX_CONTENT_WIDTH`≈860; `renderDocument` wraps the `.markdown-preview` VBox in a centering `StackPane` `.markdown-preview-wrap` that clamps to the viewport when narrower) with GitHub-like margins + vertical rhythm (padding/spacing/line-spacing tuned in `app.css`). **Keyboard scrolling** in the preview: Space/PageDown page down and Backspace/PageUp page up via a `KEY_PRESSED` filter on the preview `ScrollPane` (`scrollPreviewPage`; the pane is focused on entering PREVIEW + on click), while `C-v`/`M-v` (→ `nav.pageDown`/`nav.pageUp`) page the preview through `EditorBuffer.pagePreview` / `MainController.pageActivePreview` when the preview is the active scroll target (PREVIEW mode, or a focused SPLIT preview) and otherwise fall back to paging the editor. Rendering aims to match **GitHub**: GFM task items render as real checkboxes (the `TaskListItemMarker` is the `ListItem`'s **first child**, not nested in its paragraph — `firstTaskMarker`), inline `code` is a `Label` "pill" with a rounded gray background (a `Text` can't carry a background), `#`/`##` headings get an underline rule, and links aren't permanently underlined. Images render via `ImageView` (`imageNode`, capped to `MAX_IMAGE_WIDTH`, alt/title tooltip); a paragraph that is just an image becomes a **block** image, `http(s)`/`file`/`data:` URLs load directly and relative paths resolve against the file's folder (`resolveUrl`, `baseDir`). Loading goes through **`PreviewImageLoader`** (`editor/`): off the FX thread it fetches the bytes (URLConnection with a timeout; `data:` decoded inline) and, crucially, **rasterizes SVG** — which JavaFX's decoder can't read — via **JSVG** (`com.github.weisj.jsvg`, a small modular SVG lib; `requires com.github.weisj.jsvg` + `java.desktop`, no moditect) into a `BufferedImage` (`LoaderContext.createDefault()` denies external resource loads → no SVG-based SSRF) then copies it to a JavaFX `WritableImage` via a `PixelWriter` (no `javafx.swing`); non-SVG bytes decode with JavaFX. This is what makes **shields.io / GitHub badges** (served as `image/svg+xml`) show in the preview. Results are cached per URL and recent failures are remembered for `FAILURE_TTL_MS` so a re-render doesn't hammer an unreachable host; only the final `setImage` touches the FX thread. The pure `looksLikeSvg(bytes)` sniff is unit-tested. (JSVG can't render a nested SVG-in-SVG, e.g. a badge's embedded logo glyph — harmless; its logger is quieted to `SEVERE` so that per-render WARNING isn't spammed.) **Crucial macOS guard:** because SVG rasterization touches `java.awt`/Java2D, **`App.main` sets `java.awt.headless=true` as its very first statement**. On macOS the AWT/Java2D *native* pipeline (AppKit/Metal) otherwise contends with JavaFX's Glass/Prism for the single AppKit run loop — an **intermittent deadlock/hang** that grows likelier the more SVG badges are rasterized (i.e. the more Markdown previews are open). Headless Java2D rasterizes to a `BufferedImage` purely in software (no AppKit), so badges still render while the conflict disappears. Keep it first in `main` (before any AWT class loads); it covers every entry point (`javafx:run`, fat-jar `Launcher`, jpackage module all route through `App.main`). **Read-only / View mode**: `EditorBuffer.setViewMode/isViewMode` is a user toggle *separate* from the huge-file `setReadOnly` — `applyEditable()` sets `area.setEditable(!hugeFile && !viewMode)` and tags the surface `.read-only`, but View mode keeps highlighting/minimap/undo (unlike huge-file mode). `isEditable()` is the single source of truth. `setEditable(false)` blocks typing; the app's own edit commands are guarded by `MainController.activeEditable()` (checked in `withArea` + `onCut`/`onPaste`/`chooseLineEndings`). Command `view.toggleReadOnly` (`C-x C-q`); the state persists per file in `WorkspaceState.readOnlyFiles` (`persist/restoreReadOnly`, mirroring markdown mode), and `restoreReadOnly` also auto-enables it for files not writable on disk (pure/unit-tested `MainController.shouldOpenReadOnly`). The status bar shows a read-only **toggle** segment — "Read-Only" (amber/`.active`) ⇄ "Editable" (muted), click to flip either way — and the tab gets a `.read-only` class. `getNode()` returns an outer `BorderPane` whose top is an MS-Word-style **"View Mode" banner** (`.view-mode-bar`, shown by `updateViewModeBar()` only while `viewMode && !hugeFile`) with the navigation hint and an **Enable Editing** button — visible only when the file is writable (else a "read-only on disk" note); the button runs `onEnableEditing` (wired by `MainController.addBuffer` to `enableEditing(buffer)` → off + persist + refresh). While read-only, an `area`-level `KEY_PRESSED` filter (`addViewModePaging`, installed on `area`/`area2`) makes unmodified **Space page down / Backspace page up** (pager-style); it runs before RichTextFX's own no-op handling and only acts when `viewMode` is on. **Auto/smart indentation**: the pure, unit-tested `Indenter` (per-language `Style` BRACES/XML/PY/SHELL/RUBY/PLAIN via `styleFor`) computes Enter indentation — inherit the current line's indent, +1 level after a block opener, and a matching-pair split (`{}`/`()`/`[]`/`<a>|</a>`); `EditorBuffer.addAutoIndent` (a `KEY_PRESSED` Enter filter + a `KEY_TYPED` filter on `area`/`area2`, same pattern as the pager/snippet keys) inserts it, consuming Enter so RichTextFX's plain newline doesn't also run. Typing a closer (a `)]}` bracket alone on a line, or a completed keyword `end`/`fi`/`done`/… via `completesCloserKeyword`) re-aligns the line to its opener via `Indenter.closerAlignIndent` (nearest shallower previous line) — done on the completing keystroke, not on Enter, so it never over-dedents. Indent unit (tab vs `tabSize` spaces) is inferred from the file (`detectUnit`). **Smart backspace**: a `BACK_SPACE` `KEY_PRESSED` filter (also in `addAutoIndent`) does a one-press indent delete when the caret is in leading-only whitespace — pure/unit-tested `Indenter.smartBackspaceCount(beforeCaret, afterCaret, hasPreviousLine)`: on a **blank (auto-indented) line** it removes the indent **and the preceding newline** (so one press jumps back to the end of the previous line — "back to where you hit Enter"), on an **indented content line** it clears the indent only, and returns 0 elsewhere. The filter consumes only when the count is >1 so a normal single-char Backspace runs everywhere else. Inert while a snippet session is active or in read-only mode. **Auto-close & matching brackets**: pure, unit-tested `AutoClose` (decision for a typed char → INSERT_PAIR/SKIP_OVER/WRAP_SELECTION/NONE; quotes aren't paired next to a word char) and `BraceMatcher` (`match(text, caret, maxScan)` → the two offsets to highlight, nearest bracket left-of-caret first). `EditorBuffer.addAutoClose` (a `KEY_TYPED` filter + a Backspace `KEY_PRESSED` filter on `area`/`area2`, **registered before `addAutoClose`/`addAutoIndent` order matters** — auto-close runs first and only consumes when it acts, leaving a plain closer for the indent dedent) inserts pairs / types over / wraps / clears empty pairs. Brace matching runs off the caret listener via `scheduleBraceMatch` (coalesced to one per pulse) → `updateBraceMatch`, which adds a combinable `brace-match` style class to the two bracket chars (`.text.brace-match` in `syntax.css`, a semi-transparent box so it keeps the token color and needs no per-theme override); the syntax highlighter overwrites it on re-highlight, so the highlight apply re-calls `scheduleBraceMatch`. Off in large-file mode. **Comment/uncomment**: pure, unit-tested `Commenter` (`styleFor(language)` → `CommentStyle(line, blockStart, blockEnd)`; `toggle(text, selStart, selEnd, style)` → an `Edit`); a single line uses the line comment (else block), a multi-line selection uses the block/region comment (else line-comments each line), preserving indentation and toggling off when already commented. Command `edit.toggleComment` (`M-;`, Emacs comment-dwim) in `MainController.toggleComment`, guarded by `activeEditable()`. **Transpose**: pure, unit-tested `Transposer` (`transposeChars`/`transposeWords`/`transposeLines` → an `Edit{from,to,replacement,caret}`, or null for a no-op) implements the Emacs transpose commands; `MainController.transpose(op)` applies it to the active `CodeArea` (guarded by `activeEditable()`). Commands `edit.transposeChars` (`C-t`), `edit.transposeWords` (`M-t`), `edit.transposeLines` (`C-x C-t`). **Spell checking**: Apache **Lucene Hunspell** (pure Java; `requires org.apache.lucene.analysis.common`/`.core`, real JPMS modules → no moditect). `SpellDictionaries` builds + caches one `Hunspell` per language **lazily/off-thread** from bundled `resources/com/editora/dictionaries/<id>/index.{aff,dic}` (in-memory `ByteBuffersDirectory`); ships `en_US`/`en_GB` (SCOWL, permissive) and `es`/`fr` (used under MPL-1.1/MPL-2.0). German is GPL-only upstream, so it's not bundled. Add a language by dropping `<id>/index.{aff,dic}` under `dictionaries/` and adding the id to `SpellDictionaries.AVAILABLE` (+ NOTICE). `SpellChecker` wraps the engine + the shared user dictionary + a per-session ignore set: pure/unit-tested `wordSpans(line)` (letter runs with intra-word apostrophes) and `skip(word)` (≤1 char / digits / ALL-CAPS / camelCase) gate which words are checked; `isMisspelled`/`suggest` defer to Hunspell. Rendering is a mouse-transparent `SpellCheckOverlay` (a `Canvas`, **mirrors `WhitespaceOverlay`**) that draws red wavy underlines for visible misspelled words — decoupled from `setStyleSpans`, coalesced per pulse, visible-paragraphs only, off in large-file mode. Eligibility by applied style: prose buffers (`isProse()` = plaintext/markdown) check all words except `code`; code buffers only `comment`/`string`. The editor right-click menu (`installContextMenu`) prepends suggestions (replace the word) + **Add to Dictionary** (`ConfigManager.addUserWord` → `dictionary.txt`) + **Ignore** when over a misspelling (`spellHitAt` via `area.hit`). Toggle `view.toggleSpellCheck` (`Settings.spellCheck`, default on); per-file language `spell.setLanguage` (`QuickOpen` over `SpellDictionaries.available()`, persisted in `WorkspaceState.spellLanguages`, default `Settings.spellLanguage`). `SettingsWindow` also has an "Enable spell check" checkbox and a default-language `ComboBox` (friendly names via `spellLanguageName`), applied live through the standard `apply()`/`applyViewSettingsToAllBuffers` path. Syntax highlighting uses **TextMate grammars** via tm4e: `GrammarRegistry` (singleton) maps a file extension to a bundled `.tmLanguage.json` grammar under `resources/com/editora/grammars/`; `TextMateHighlighter` tokenizes the document line-by-line (carrying grammar state across lines) and maps each token's TextMate scope to a CSS class themed in `styles/syntax.css`. Files without a bundled grammar are left unstyled. `LanguageRegistry` is now only an extension→language-name resolver used by `FoldRegions`/`FoldManager` (fold strategy) and the File Information panel — it no longer does highlighting. **Grammar resources live in the `com.editora.grammars` package, which `module-info.java` must `opens ... to org.eclipse.tm4e.core`; without that, JPMS encapsulation makes tm4e's `Class.getResourceAsStream` return null at runtime (module path) and grammars silently fail to load even though classpath tests pass.**
- `snippet/` — VS Code/TextMate **snippets**. `SnippetParser` (pure/unit-tested) parses a body into a `ParsedSnippet` (literal `text` + `TabStop`s with relative offset ranges): `$1`/`${1:default}`/`$0`, mirrors (repeated number), `${1|a,b|}` choices (captured per stop; `SnippetSession` shows a `ContextMenu` dropdown when the field is entered, first option is the default text), `$VAR`/`${VAR:default}` resolved by `VariableResolver` (`TM_FILENAME`/`TM_FILENAME_BASE`/`TM_DIRECTORY`/`TM_FILEPATH`, selection, `CLIPBOARD`, date/time, line vars), `\$ \} \\` escapes; transforms `${1/re/fmt/}` are parsed-and-ignored. `SnippetManager` loads bundled `resources/com/editora/snippets/<lang>.json` + `global.json` and user `<configDir>/snippets/<lang>.json` (user overrides bundled by prefix; language overrides global), cached until `reload()`. **Bundled snippets cover all 21 highlighted languages** — 15 vendored unmodified from the MIT [friendly-snippets](https://github.com/rafamadriz/friendly-snippets) collection (attributed in `NOTICE`), the other 6 (xml/batchfile/groovy/json/yaml/ini) + `global` written for Editora. The Jackson reader is **lenient for real-world VS Code files**: `FAIL_ON_UNKNOWN_PROPERTIES=false` (ignores `scope`), and `prefix`/`body`/`description` each accept a string *or* an array (array `prefix` registers every trigger). **No `module-info` opens needed for the resources** — our own code reads them via `getResourceAsStream` (same module), like the keymap; only the Jackson DTO needs `opens com.editora.snippet to com.fasterxml.jackson.databind`. `SnippetSession` (in `editor`, bound to the `CodeArea`) inserts the expansion (re-indenting continuation lines to the caret's indent), tracks absolute field ranges through edits via a `plainTextChanges` subscription using the pure `SnippetSession.shift(...)`/`reindent(...)` (mirror of `BookmarkManager.shift`), mirrors the active field live (an `applying` reentrancy guard), and steps stops with Tab/Shift-Tab to `$0`; `$0` is tracked as a shifting range so it lands correctly after earlier-field edits. `EditorBuffer` holds the session, installs `addSnippetKeys` (Tab/Shift-Tab/Escape filter on `area`/`area2`, only consuming Tab when a session is active or a prefix matched — else indent), and gets the lookup via an injected `setSnippetProvider` (decoupled from `SnippetManager`, like `gutterBookmarkClick`). ## Conventions

- **Performance is a top priority.** Editora must stay fast and responsive, especially on large files. Treat the UI thread as sacred and these as the hot paths: typing/editing, scrolling, syntax highlighting, the document overlays (whitespace, minimap, 80-column ruler), and the line-number gutter. Rules to uphold:
  - **Never block the JavaFX Application Thread** with non-trivial work — tokenize/parse/search off-thread (see the `highlightExecutor` pattern in `EditorBuffer`) and apply results back on the FX thread with a generation/stale-result guard.
  - **Debounce and coalesce** — re-highlighting is debounced; overlay/ruler/minimap redraws coalesce to one per pulse (a `pending` flag + `Platform.runLater`). Don't add per-keystroke or per-scroll-pulse work that isn't coalesced.
  - **Work incrementally / only on what's visible** — highlighting re-tokenizes only from the changed line; overlays iterate just the visible paragraphs (`firstVisibleParToAllParIndex … last`) and skip folded lines. Avoid O(document) work on edits or scrolls.
  - **Don't defeat the per-node CSS style cache** — keep token rules as the compound `.text.<class>` selector (see `syntax.css`); coalesce adjacent same-style spans (`SpanMerger`) before `setStyleSpans`; never query `getCharacterBoundsOnScreen` synchronously inside a layout/viewport event.
  - **Large/huge files** already disable highlighting + minimap (≥5 MB) and go read-only with a capped load (≥50 MB); preserve those guards. Bound memory (undo history is capped; loads are capped).
  - **Bound retained GPU textures.** JavaFX's Prism texture pool has a fixed ceiling (default 512 MB); exhausting it makes the render thread NPE (null `RTTexture`/mask texture) → a black window — seen only in the packaged build. So don't let GPU-backed resources grow with the number of open files: a background (non-selected) tab's buffer drops its minimap snapshot via `EditorBuffer.setRenderingActive(false)` (driven from the tab-selection + tabs-added listeners; `Minimap.setRenderingActive` nulls the cached snapshot and stops rendering, regenerating on show), and the `PreviewImageLoader`/`MermaidImages` image caches are **LRU-bounded** (not unbounded `ConcurrentHashMap`s) since each holds an `Image` (a pinned texture). The `dist` jpackage app (and `mvn javafx:run`) also raise the caps as a safety net (`-Dprism.maxvram=2G -Dprism.maxTextureSize=16384`). When adding any per-buffer Canvas/Image, make sure it's released or bounded.
  - **Assess and report the performance cost of every change.** For any implementation or bug fix, evaluate its cost on the hot paths (allocation per keystroke/scroll, added FX-thread work, extra layout/CSS passes, memory) and tell the user about it — even if it's "negligible." If a change risks a regression, measure (e.g. the temporary `System.nanoTime` instrumentation used for the startup investigation) rather than guess.
- **Adding a feature = add a `Command`, and surface it in the command palette.** Every time we add a command or an applicable user-facing feature, it must be discoverable in the palette. Register it in `MainController.registerCommands()` and, if it needs a key, add a binding to `emacs.json`. The palette is populated from `CommandRegistry.all()`, so a properly registered command appears automatically — never add a user-facing action that bypasses the registry. Toolbar buttons and the palette both dispatch through commands; don't wire UI handlers to logic directly when a command fits.
- **Every user-facing string is localized — never hardcode UI text.** Any new label, status/echo message, tooltip, dialog title/header/body, button, menu item, prompt, or placeholder must go through the i18n catalog: `import static com.editora.i18n.Messages.tr;` then `tr("some.key")` (or `tr("key", args…)` for `MessageFormat` parameters like `Saved {0}`) — never pass a raw English literal to a JavaFX control. **Add the key to all six `resources/com/editora/i18n/messages[_<lang>].properties` files** (English base + `it/es/fr/pt/de`) with a real translation in each; `MessagesTest`'s key-parity check fails the build if any locale is missing a key. New `Command`s get their title for free via `Command.of(id, runnable)` + a `command.<id>` key. Only deliberate exceptions stay literal: technical tokens (`UTF-8`, `LF`/`CRLF`, `Ln`/`Col`, git verbs in `gitlabel.*`, detected language names, example URLs) and the pre-GUI `--help`/`--version` text. See the `i18n/` architecture bullet for the full system.
- **App icon / branding:** the source logo is `branding/editora-icon.svg`. Window-icon PNGs (16–512) live in `resources/com/editora/icons/` (loaded into `stage.getIcons()` in `App`; also the About-dialog logo). Native-installer icons `branding/editora.{icns,ico,png}` are generated from the SVG (rsvg-convert + iconutil + a small ICO writer) and passed to jpackage via `${jpackage.icon}`, set per-OS in the OS-activated profiles. Regenerate after editing the SVG.
- **Bundled fonts:** five monospace families ship under `resources/com/editora/fonts/<family>/` (Regular/Bold/Italic/BoldItalic TTFs; Fira Code has no italic), **plus Inter** (sans-serif, `fonts/inter/`) as the Markdown **preview + PDF prose** font. `Fonts.load()` registers them all via `Font.loadFont` at the very start of `App.start` (before any CSS), so they resolve by family name regardless of the OS. `Fonts.BUNDLED` (JetBrains Mono first = default) is the **monospace** editor-font picker list and deliberately **excludes Inter** (Inter isn't an editor font); it's listed ahead of the system monospaced families in the Settings font picker. Inter is the preview prose font on **all platforms** (`.markdown-preview .md-text` in `app.css` — pinned explicitly because raw `Text` nodes don't reliably inherit the scene font, and because the macOS chrome's logical "System" family can't be bold-rasterized cleanly by JavaFX) and the PDF prose font (`MarkdownPdfWriter`). They're plain resources (no `module-info`/moditect change) and are picked up by the `dist` build automatically. Attributed in `NOTICE` (all OFL-1.1).
- **Theming:** AtlantaFX is the user-agent stylesheet and themes standard controls (`Themes`, set via `Application.setUserAgentStylesheet`). It does **not** style the RichTextFX `CodeArea` — editor surface, line-number gutter, and syntax token colors live in `styles/app.css` + `styles/syntax.css` (the **Primer Light** defaults). **Editor color themes** (`EditorThemes`) layer an override stylesheet from `styles/editor-themes/<name>.css` onto the scene *after* those defaults (Primer Light = no override); `MainController.applyEditorTheme` swaps that sheet on the scene and sets each buffer's current-line highlight color (RichTextFX's line-highlight fill is set in code, not CSS). The editor theme follows the AtlantaFX theme (`EditorThemes.defaultFor`) until the user picks one (`Settings.editorThemeUserSet`). Token rules must stay the compound `.text.<class>` form — see the header comment in `syntax.css`. The **Project tool window file tree** is themed via two looked-up CSS colors `-project-folder-color`/`-project-file-color` (defined on `.project-tree` in `app.css`, defaulting to AtlantaFX accent/muted); each editor theme overrides them, so a new editor theme should add a `.project-tree { -project-folder-color: …; -project-file-color: …; }` rule.
- **Tests** cover pure logic only (keymap resolution, command registry, config merge, highlighting spans). There is no GUI/toolkit test harness; verify interactive behavior by running the app.

## Packaging note (important)

RichTextFX and its transitive deps (`reactfx`, `flowless`, `undofx`, `wellbehavedfx`)
are **automatic modules**, which `jlink` cannot link. The `moditect-maven-plugin` in
the `dist` profile injects explicit `module-info` descriptors into them. If you bump
RichTextFX or add a dep it uses, you may need to adjust those descriptors' `requires`
(e.g. several of them need `javafx.controls` for `IndexRange`). Use RichTextFX
**0.11.7+** — earlier versions are incompatible with JavaFX 25.

tm4e (the syntax engine) ships as the NetBeans repackaging
`org.netbeans.external:org.eclipse.tm4e.core-0.14.0:RELEASE260` (tm4e is not on
Maven Central). Its Oniguruma backend (`org.jruby.joni:joni`, `org.jruby.jcodings:jcodings`)
and `com.google.code.gson:gson` are already proper modules, but tm4e core is an
automatic module, so `moditect` injects a `module-info` for it too. The NetBeans
jar is also **code-signed**, and `jlink` rejects signed modular jars — the `dist`
profile's antrun step strips `META-INF/*.SF,*.RSA,*.DSA,*.EC` before linking.
