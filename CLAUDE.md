# Editora

A keyboard-driven, cross-platform programmer's text editor. JDK 25 + JavaFX 25 +
Maven, modular (JPMS, module `com.editora`).

## Commands

- Run the app: `mvn javafx:run`
- Run tests: `mvn test`
- Build app image / native installer: `mvn -Pdist package`
  - Produces `target/dist/Editora.app` (macOS); OS profiles auto-select DMG/MSI/DEB.
  - Quick unpackaged bundle: `mvn -Pdist -DskipTests -Djpackage.type=APP_IMAGE package` ⇒ `target/dist/Editora.app`.
  - **Both** also run the AOT-cache training step (a GUI window flashes for ~2.5 s, then the build
    injects `editora.aot`) — see the AOT-cache note under *Conventions → performance*. It's
    failure-tolerant, so on a display-less machine it just skips the cache.
- Build a runnable fat jar: `mvn -Pfatjar package` ⇒ `target/Editora-<version>.jar`, run with
  `java -jar`. It bundles JavaFX (classes + natives) for **the build host's platform only** and runs
  from the classpath via the non-`Application` `com.editora.Launcher` main class. A single
  all-platforms jar is impossible (JavaFX's macOS/Linux x64 and arm64 natives share filenames and
  collide), so the release CI builds one fat jar per runner.
- Cut a release: bump `<version>` in `pom.xml`, push a `vX.Y.Z` tag (`-rcN` ⇒ pre-release).

Run Maven from the project root (`/Users/adriandeleon/src/adl/Editora-V2`).

## Worktrees — one per task (important)

This repo is worked on by multiple Claude Code sessions. **Each task gets its own
`git worktree`** so sessions don't share a single working tree (which previously caused commits to
land on the wrong branch when a parallel session ran `git checkout` in the shared checkout).

- **Start a task:** `scripts/worktree.sh new <branch>` (e.g. `feat/foo`, base defaults to
  `origin/master`). It creates an isolated checkout at `../Editora-V2-worktrees/<slug>` on a new
  branch; `cd` into it and do the work there. `list` shows all worktrees; `rm <branch>` removes the
  worktree + branch after merging; `prune` cleans stale metadata.
- **Never `git checkout` a different branch in the main checkout** (`/Users/adriandeleon/src/adl/Editora-V2`)
  while other sessions may be active — switch by creating/entering a worktree instead. Keep the main
  checkout on `master`.
- **Subagents:** spawn them with the Agent tool's `isolation: "worktree"` so background work can't
  disturb the main checkout.
- **A session/terminal does not "move into" a worktree by creating one.** A session is anchored to
  the directory it launched in; a worktree is just a separate folder. To actually *work inside* one
  interactively, open a **new terminal / Claude Code session** in that directory
  (`cd ../Editora-V2-worktrees/<slug>`). A long-running session pinned to the main checkout (e.g. the
  one whose terminal sits on `master`) instead operates on a worktree **by path** — absolute paths for
  edits, `cd <worktree> && ./mvnw …` for builds — and stays on `master`; commit/merge from there but
  **never `git checkout` another branch in the main checkout**. (Claude's Bash tool also resets its
  working dir to the project root after each command, so it can only reach a worktree by path, not by
  persistently `cd`-ing in.)
- Worktrees live in the sibling `../Editora-V2-worktrees/` dir (outside the repo), so nothing needs
  to be gitignored. They share the same `.git`, so all branches/commits are visible everywhere.

## Release pipeline

`.github/workflows/release.yml` runs on a `v*` tag (or manual dispatch for a dry run): a 5-way
matrix (linux x64/arm64, macOS x64/arm64, windows x64 — **Windows arm64 is omitted: a hosted
runner now exists (`windows-11-arm`, GA Jan 2026), but OpenJFX 25 publishes no `win-aarch64` native
jar on Maven Central (see [JDK-8314064]), so a native ARM64 build can't link — Windows-on-ARM users
run the x64 installer under x64 emulation. Revisit when JavaFX ships win-aarch64 natives.** macOS x64
uses the `macos-15-intel` runner — the last Intel x86_64 image (good through ~Aug 2027) — since the
old `macos-13` Intel runner was retired Dec 2025;
each on its own GitHub-hosted runner) builds the native
installer via the existing `-Pdist` profile — there is **no cross-building** (jpackage + JavaFX are
host-specific), so each runner builds for itself. Each runner also builds a per-platform runnable
fat jar via `-Pfatjar` (`Editora-<version>-<target>.jar`). Installers are renamed to
`Editora-<version>-<target>.<ext>` per target — one consistent `Editora-<version>-<target>` prefix
across all artifacts (jpackage's DMG/MSI names omit the version + arch; the
version comes from a `Resolve version` step — the tag minus `v`, else the pom version) and uploaded as
artifacts alongside the fat jar; a final job
hands them to **JReleaser** (`jreleaser.yml`, via `jreleaser/release-action`) which creates the
GitHub release with all installers + fat jars + `checksums.txt` + a changelog. JReleaser only *orchestrates the release* — it does not
build (the `dist` profile is reused as-is) and there is **no `pom.xml`/Maven change**, so the normal
build is unaffected. Installers are currently **unsigned** (signing/notarization is a follow-up).
Uses the BellSoft **Liberica** JDK 25 in CI for full arch coverage (incl. linux aarch64).

## Architecture

Single module `com.editora` (see `src/main/java/module-info.java`). `App` is the
JavaFX entry point: it loads the **shared** config (`config/SharedConfig`), localizes the UI, applies
the theme + keymap, then hands off to **`ui/WindowManager`** which builds the window(s) — **the app is
multi-window** (each project opens in its own top-level window; see the `ui/`/`config/` bullets). `App`
also pins a classloader before building any window: `FXMLLoader.setDefaultClassLoader(...)` +
`Thread.currentThread().setContextClassLoader(...)`, because on macOS the FX/AppKit thread's context
classloader can be null, which would otherwise NPE inside `FXMLLoader.load()` when a window is built at
runtime (and break lazy class loading on that thread). `WindowManager.buildWindow` is the per-window
construction (extracted from `App.start`): it loads config, builds the command + keymap system, installs
the key dispatcher on the scene, applies the theme, and loads `ui/main.fxml`. It
wraps the FXML root in a `StackPane` so a floating overlay can sit on top — used by
`MainController.installZenOverlay` for the **Zen-mode exit button** (a small floating "Z",
`Icons.zen()`, styled `.zen-exit` like the Markdown preview controls, shown top-right only while
`WorkspaceState.isZenMode()` — toggled in `applyChromeVisibility` via `updateZenButton` — and
clicking it runs `setZenMode(false)`). `updateZenButton` also drops the Z below the Markdown preview
controls (a larger top margin) when the active buffer is Markdown, since those also float top-right; it
re-runs on tab switches so the offset tracks the active file. `installZenOverlay` also creates a second
floating button, the **toolbar-restore "Tool" button** (`Icons.tools()`, styled `.toolbar-restore` like
the Zen "Z", also `Pos.TOP_RIGHT` — it never coexists with the "Z", which only shows in Zen mode), shown by `updateToolbarRestoreButton` (called from
`applyChromeVisibility`) only when the toolbar is hidden and not in Zen (`!isShowToolbar() && !isZenMode()`);
clicking it runs `toggleToolbar()` (which also `settingsWindow.syncToolbarCheck()`s), so a user who hides
the toolbar (Settings checkbox or the `view.toggleToolbar` palette command) can always bring it back. The
**toolbar's state-aware edit icons** (Save/Undo/Redo/Cut/Copy/Paste) are kept enabled/disabled by
`refreshEditState()` (save←dirty, undo/redo←`area.isUndoAvailable/isRedoAvailable`, cut←selection+editable,
copy←selection — all disabled on a non-buffer/Welcome tab) and `refreshPasteState()` (paste←editable +
`Clipboard.hasString()`, kept off the per-keystroke path: run on tab switch, window focus-regain, and after
cut/copy). `refreshEditState` runs on tab switch and the active buffer's edit/selection/dirty pulses;
disabling is cosmetic only — the `C-s`/`C-z`/… keymap commands still work. The toolbar's **Find in Files**
icon (`Icons.findInFiles()`, `onFindInFiles → openSearchInFiles`) sits beside the in-file Find icon.

- `command/` — the keyboard-driven core.
  - `Command` / `CommandRegistry`: every action is a registered `Command` (id + title + runnable). The registry feeds both keybindings and the palette.
  - `KeymapManager`: maps chord sequences (e.g. `C-x C-s`) to command ids; `loadNamed(name)` loads the bundled `resources/com/editora/keymaps/<name>.json`, then `applyOverrides` layers user (and plugin) overrides. **Five bundled keymaps** via the static ordered `KeymapManager.AVAILABLE` (id→display name): `emacs` (default), `cua`, `sublime`, `vscode`, `intellij` — the four non-Emacs ones are **non-modal** (just different chord→command maps over the existing command ids, so they fit the flat resolver; the commands are all palette-reachable, so a keymap only changes accelerators and never strands functionality). **Per-OS files:** each GUI keymap ships a base `<name>.json` (Ctrl-based, Win/Linux) **and** a complete `<name>.mac.json` (Cmd-based) — `loadNamed` prefers the `.mac` variant on macOS (full replacement, not a merge, so no Ctrl-shadowing or modifier token-order bug); Emacs is single-file (Control on every platform). Chord tokens are exactly what `KeyDispatcher.chord()` emits, modifiers in canonical order `C- M- Cmd- S-`; **function keys work via the `keyName` fallback** (`KeyCode.F5`→`"f5"`), so VSCode/IntelliJ F-key bindings need no engine change. **Live switching:** the Settings → Keymaps picker (the formerly-placeholder category) + the `keymap.select` palette command (`MainController.chooseKeymap`→`applyKeymap`) set `Settings.keymap`, then `WindowManager.reloadSharedKeymap()` rebuilds the **single shared** `KeymapManager` (base→user overrides→enabled plugins' `manifest.keymap`, mirroring `applyPlugins`) and `broadcastSettingsApplied()`s — every window's `KeyDispatcher` reads the same instance, so the switch is instant with no restart (a stale mid-chord prefix self-cancels on the next key). Each window's `reapplyAfterSharedSettingsChange` then **refreshes every chord-derived hint** so none stays frozen to the old keymap: **toolbar tooltips** (`setupButton(button, icon, tip, commandId)` → `applyToolbarTip`/`refreshToolbarTooltips` strip the chord baked into the `tooltip.*` i18n string — `"Save (C-x C-s)"` → `"Save"` — and append the live `invertBindings().get(id)`), the **command palette** (`CommandPalette.refreshBindings()`), the **tool-window stripe tooltips** (`ToolWindowManager.refreshTooltips()`), and the **Welcome shortcut labels** (`WelcomePane.refresh()`'s `shortcutFor`). The status-bar read-only/git-switch tooltips had their baked Emacs chords stripped from i18n. `KeymapsTest` guards all bundled keymaps (parse, valid command ids vs the `command.*` i18n keys, canonical token order, base↔`.mac` same-command-set parity). **Keybinding editor (fully configurable shortcuts):** Settings → Keymaps (below the theme combo) lists every command (from `CommandRegistry.all()`) with its current chord and a filter; a **Record** button turns a row into a live capture field (`KeyDispatcher.chord(e)` is now `public`; space-joins multi-key sequences, Esc cancels), **Reset** restores one command's default, and **Reset all** clears every override. The mutation logic is the pure, unit-tested **`command/KeybindingEdits`** (`rebind`/`clear`/`reset` over `(baseMap, overrides)`): a rebind drops the command's prior user entries, suppresses each base default via the **`KeymapManager.UNBIND`** sentinel (a blank override value → `applyOverrides` *removes* that chord), then binds the new chord; `reset` removes the command's user entries + its default-suppressors. `MainController` wires the editor (`SettingsWindow.ShortcutActions`/`Shortcut`): `shortcutRows()` (registry + `invertBindings`), `baseBindings()` (a fresh `KeymapManager.loadNamed` with no overrides), conflict check via `keymap.commandFor`, and `rebindShortcut/resetShortcut/resetAllShortcuts` → persist `Settings.keybindings` + `reloadKeymap()` (live, all windows; overrides are global and layer on top of the active theme). The recorder runs in the Settings window's own scene (no `KeyDispatcher` there), so capture is clean. **New editor commands** `edit.selectAll` + the pure/unit-tested **`editor/LineOps`** (`duplicateLine`/`moveLineUp`/`moveLineDown` → an `Edit{from,to,replacement,caret}`, mirroring `Transposer`; applied by `MainController.lineOp(...)` guarded by `activeEditable()`) are bound in the GUI keymaps (Ctrl/Cmd-A, Alt/Option-Up/Down, etc.; Emacs leaves them palette-only since `C-a` is `nav.lineStart`). *Deferred: modal **Vim** (needs a normal/insert/visual mode state machine, operator-pending, counts, registers, `:` command line — the flat resolver can't express it); the keybinding recorder allows a modifier-less chord (can shadow typing) and multi-line-selection block move/duplicate (v1 acts on the caret's line).*
  - `KeyDispatcher`: a scene-level `KEY_PRESSED` filter that builds chord tokens and resolves them, holding a pending-prefix buffer for multi-key Emacs chords. Lone unbound keys are *not* consumed, so normal typing works. **Windows/Linux Alt menu-mode fix (confirmed via on-device key logging):** on Windows a bare `Alt` **or an unbound `Alt+<key>`** is treated by the OS as menu/mnemonic activation, which puts the native window into "menu mode" — that freezes `KEY_TYPED` app-wide (mouse still works, restart required) and breaks the many `M-` chords (`M-x`, `M-g`, `M-1`…`M-9`, …). So on non-macOS `handle()` consumes **(a)** a bare `Alt` press and **(b)** any *unbound* key while plain Alt is held, and `install()` adds a `KEY_RELEASED` filter that consumes the bare `Alt` release. Bound `M-` chords still run+consume normally. The plain-vs-AltGr test is the pure, unit-tested **`plainAltActive(isMac, altDown, controlDown)`** = `!isMac && altDown && !controlDown` — **AltGr** (reported as Ctrl+Alt) and Ctrl+Alt chords are deliberately **not** consumed, so international layouts keep composing characters. macOS is unaffected (Option = Meta). *(Note: a global hotkey manager on the user's machine — e.g. XKeymaps — can still grab a specific chord like `Alt+X` before Editora sees it; that's external, not an Editora bug.)*
- `editor/` — `EditorBuffer` wraps a RichTextFX `CodeArea` (line numbers + fold chevrons via `FoldManager`, debounced highlighting). **Bookmarks**: a per-buffer `BookmarkManager` (analogous to `FoldManager`) holds line→`Bookmark` (record `line/note/lineText` in `config`), tracks lines through edits via `plainTextChanges` so bookmarks **follow their content** with forward/"sticky" gravity — inserting at a bookmarked line's start pushes it down (pure, column-aware `BookmarkManager.shift(...)` is unit-tested). Because `shift` only runs for edits made *inside* the app, on open each bookmark is **re-anchored to its saved `lineText`** (pure, unit-tested `BookmarkManager.reanchor(...)`: keeps an exact match in place, else moves it to the *nearest* line whose stripped text matches — downward-first on ties — bounded by `MAX_REANCHOR_SCAN`; an empty/vanished `lineText` stays at its clamped stored line), so bookmarks survive **external** edits to the file. `restore(...)` returns whether anything moved and `MainController.restoreBookmarks` then calls `persistBookmarks` to self-heal the saved indices once (later opens hit the exact-match fast path, no save). The gutter draws a `.bookmark-marker` glyph (wired through `FoldManager.setBookmarkHooks`, refreshed per-line via `recreateParagraphGraphic`). Bookmarks live in their own `bookmarks.json` (`config/BookmarkStore`) — kept out of `WorkspaceState` so they're all in one easy-to-find file — but **scoped per project**: the store is `Map<projectKey, Map<path,List<Bookmark>>>` (`byProject`; `projectKey` = `""` for the no-project/global session, else the project id). `ConfigManager.getBookmarks()` returns the **active** project's bucket, chosen by `currentBookmarkKey()` (derived from the active `workspaceStateFile`: `workspace-state.json` → `""`, `projects/<id>.json` → `<id>`), so switching a project (`setWorkspaceStateFile`/`useDefaultWorkspaceStateFile`) automatically swaps which bookmarks are visible — `MainController.activateSession` already calls `bookmarksPanel.refresh()`. `saveBookmarks()` writes the whole `bookmarks.json`; `MainController.persistBookmarks`/`restoreBookmarks` call `saveBookmarks()` — *not* `save()`. On first run, `ConfigManager.loadBookmarks()` does a one-time migration: it pulls bookmarks previously embedded in `workspace-state.json` (→ the `""` bucket) and each `projects/<id>.json` (→ that project's bucket), strips the legacy `bookmarks` node from those files, and writes `bookmarks.json` (created even when empty) so the migration runs once. Deleting a project also drops its bookmarks (`MainController.deleteProject` → `ConfigManager.deleteBookmarksForProject(id)`). The `BookmarksPanel` reads the active bucket (so it includes closed files) and routes edits back through the controller (open buffer's manager, else the map). **Order is user-controlled, not sorted**: the panel renders the bucket's map order (files) and each file's list order (bookmarks) verbatim, and the `M-g b` jump picker (`allBookmarkEntries`) reads the same order — so the two always agree. Reorder a bookmark within its file or a whole file group via **Alt+Up/Down**, a right-click **Move Up/Down**, or **drag-and-drop** (bookmark onto a sibling, file header onto another; disabled while a filter is active), routed through `Actions.moveBookmark`/`moveFile` → `MainController.moveBookmark`/`moveBookmarkFile` (mutate the bucket + `saveBookmarks`; the panel reselects the moved row after refresh). The drag uses the **same visuals as the tab strip**: a translucent `setDragView` snapshot follows the cursor, the dragged `.tree-cell` dims (`.bookmark-dragging`), and an accent insertion line shows on the target row's top/bottom edge (`.bookmark-drop-above`/`.bookmark-drop-below`, mirroring `.tab-drop-before/after`); `BookmarksPanel.insertIndex` (pure/unit-tested) maps the drop side to the post-removal insert index. Because an open buffer's `BookmarkManager.snapshot()` is line-ordered, `persistBookmarks` runs the pure, unit-tested `BookmarkStore.mergePreservingOrder(previous, snapshot)` to keep the custom order across edits (matching bookmarks by line, else by captured `lineText` so the order follows content through line shifts). Commands: `bookmarks.toggle` (`C-c m`), `bookmarks.next`/`previous` (`C-c ]`/`C-c [`, within-file), `bookmarks.jump` (`M-g b`, cross-file picker), `editNote`/`clearFile`. **Markdown preview**: an IntelliJ-style per-buffer 3-mode view (`EditorBuffer.MarkdownViewMode` EDITOR/SPLIT/PREVIEW) driven by a floating `MarkdownViewToggle` overlaid top-right of the editor for Markdown files; `MarkdownRenderer` parses CommonMark+GFM (commonmark-java, all proper JPMS modules — `requires org.commonmark.*`, no moditect) **off-thread** then builds native JavaFX nodes **on the FX thread** under a stale-gen guard (the `highlightExecutor` idiom), debounced via `multiPlainChanges().successionEnds(250ms)`, only while SPLIT/PREVIEW (≥5 MB files render once). The mode is persisted per file in `WorkspaceState.markdownViewModes` (`MainController.persistMarkdownMode`/`restoreMarkdownMode`), mirroring folds. Preview colors use AtlantaFX semantic CSS (`.markdown-preview`/`.md-*` in `app.css`) so they track the theme without per-theme overrides. **Independent light/dark preview theme**: the preview can be themed separately from the app/editor theme via a floating sun/moon button added to the `−`/`+` zoom control (top-left). Persisted globally in `Settings.markdownPreviewTheme` (`""` = follow the app theme until first toggled, then `"light"`/`"dark"`; schema bumped 2→3, additive identity migration). `MainController.toggleMarkdownPreviewTheme` (also the palette `view.toggleMarkdownPreviewTheme`) flips the effective theme — seeded from app brightness via `appThemeDark()` (`Themes.backgroundFor`) — saves, and `applyMarkdownPreviewTheme()` pushes it to every open buffer. `EditorBuffer.applyPreviewTheme(mode, appDark)` adds `md-light`/`md-dark` to the preview `ScrollPane`, which **redefine the looked-up `-color-*` vars for the whole preview subtree** so the existing `.md-*` rules re-resolve to a fixed GitHub-style palette (no rule duplication), and refreshes the toggle glyph (sun while dark → switch to light, moon while light → switch to dark). The toggle is injected via `setPreviewThemeToggle` (like the snippet/completion providers — `editor` can't depend on `ui`). No Settings-window entry. For readability the content is a **centered, width-capped column** (`MAX_CONTENT_WIDTH`≈860; `renderDocument` wraps the `.markdown-preview` VBox in a centering `StackPane` `.markdown-preview-wrap` that clamps to the viewport when narrower) with GitHub-like margins + vertical rhythm (padding/spacing/line-spacing tuned in `app.css`). **Keyboard scrolling** in the preview: Space/PageDown page down and Backspace/PageUp page up via a `KEY_PRESSED` filter on the preview `ScrollPane` (`scrollPreviewPage`; the pane is focused on entering PREVIEW + on click). The preview `ScrollPane` also has a **right-click menu** (`setOnContextMenuRequested`→`showPreviewContextMenu`): **Select All** / **Copy** (both → `copyPreviewToClipboard` → `MarkdownRenderer.plainText(getContent())`, the commonmark `TextContentRenderer` rendering of the source = the visible text with markup stripped; a diagram copies its source) + **Export to PDF** / **Print** (injected `setPreviewExportPdfHandler`/`setPreviewPrintHandler` → `MainController.exportPreviewPdf`/`printPreview`). The native preview nodes aren't selectable, so copy is **menu-driven** (no drag-highlight). While `C-v`/`M-v` (→ `nav.pageDown`/`nav.pageUp`) page the preview through `EditorBuffer.pagePreview` / `MainController.pageActivePreview` when the preview is the active scroll target (PREVIEW mode, or a focused SPLIT preview) and otherwise fall back to paging the editor. Rendering aims to match **GitHub**: GFM task items render as real checkboxes (the `TaskListItemMarker` is the `ListItem`'s **first child**, not nested in its paragraph — `firstTaskMarker`), inline `code` is a `Label` "pill" with a rounded gray background (a `Text` can't carry a background), `#`/`##` headings get an underline rule, and links aren't permanently underlined. Images render via `ImageView` (`imageNode`, capped to `MAX_IMAGE_WIDTH`, alt/title tooltip); a paragraph that is just an image becomes a **block** image, `http(s)`/`file`/`data:` URLs load directly and relative paths resolve against the file's folder (`resolveUrl`, `baseDir`). Loading goes through **`PreviewImageLoader`** (`editor/`): off the FX thread it fetches the bytes (URLConnection with a timeout; `data:` decoded inline) and, crucially, **rasterizes SVG** — which JavaFX's decoder can't read — via **JSVG** (`com.github.weisj.jsvg`, a small modular SVG lib; `requires com.github.weisj.jsvg` + `java.desktop`, no moditect) into a `BufferedImage` (`LoaderContext.createDefault()` denies external resource loads → no SVG-based SSRF) then copies it to a JavaFX `WritableImage` via a `PixelWriter` (no `javafx.swing`); non-SVG bytes decode with JavaFX. This is what makes **shields.io / GitHub badges** (served as `image/svg+xml`) show in the preview. Results are cached per URL and recent failures are remembered for `FAILURE_TTL_MS` so a re-render doesn't hammer an unreachable host; only the final `setImage` touches the FX thread. The pure `looksLikeSvg(bytes)` sniff is unit-tested. (JSVG can't render a nested SVG-in-SVG, e.g. a badge's embedded logo glyph — harmless; its logger is quieted to `SEVERE` so that per-render WARNING isn't spammed.) **Crucial macOS guard:** because SVG rasterization touches `java.awt`/Java2D, **`App.main` sets `java.awt.headless=true` as its very first statement**. On macOS the AWT/Java2D *native* pipeline (AppKit/Metal) otherwise contends with JavaFX's Glass/Prism for the single AppKit run loop — an **intermittent deadlock/hang** that grows likelier the more SVG badges are rasterized (i.e. the more Markdown previews are open). Headless Java2D rasterizes to a `BufferedImage` purely in software (no AppKit), so badges still render while the conflict disappears. Keep it first in `main` (before any AWT class loads); it covers every entry point (`javafx:run`, fat-jar `Launcher`, jpackage module all route through `App.main`). **Read-only / View mode**: `EditorBuffer.setViewMode/isViewMode` is a user toggle *separate* from the huge-file `setReadOnly` — `applyEditable()` sets `area.setEditable(!hugeFile && !viewMode)` and tags the surface `.read-only`, but View mode keeps highlighting/minimap/undo (unlike huge-file mode). `isEditable()` is the single source of truth. `setEditable(false)` blocks typing; the app's own edit commands are guarded by `MainController.activeEditable()` (checked in `withArea` + `onCut`/`onPaste`/`chooseLineEndings`). Command `view.toggleReadOnly` (`C-x C-q`); the state persists per file in `WorkspaceState.readOnlyFiles` (`persist/restoreReadOnly`, mirroring markdown mode), and `restoreReadOnly` also auto-enables it for files not writable on disk (pure/unit-tested `MainController.shouldOpenReadOnly`). The status bar shows a read-only **toggle** segment — "Read-Only" (amber/`.active`) ⇄ "Editable" (muted), click to flip either way — and the tab gets a `.read-only` class. `getNode()` returns an outer `BorderPane` whose top is an MS-Word-style **"View Mode" banner** (`.view-mode-bar`, shown by `updateViewModeBar()` only while `viewMode && !hugeFile`) with the navigation hint and an **Enable Editing** button — visible only when the file is writable (else a "read-only on disk" note); the button runs `onEnableEditing` (wired by `MainController.addBuffer` to `enableEditing(buffer)` → off + persist + refresh). While read-only, an `area`-level `KEY_PRESSED` filter (`addViewModePaging`, installed on `area`/`area2`) makes unmodified **Space page down / Backspace page up** (pager-style); it runs before RichTextFX's own no-op handling and only acts when `viewMode` is on. **Auto/smart indentation**: the pure, unit-tested `Indenter` (per-language `Style` BRACES/XML/PY/SHELL/RUBY/LUA/PLAIN via `styleFor`) computes Enter indentation — inherit the current line's indent, +1 level after a block opener, and a matching-pair split (`{}`/`()`/`[]`/`<a>|</a>`); `EditorBuffer.addAutoIndent` (a `KEY_PRESSED` Enter filter + a `KEY_TYPED` filter on `area`/`area2`, same pattern as the pager/snippet keys) inserts it, consuming Enter so RichTextFX's plain newline doesn't also run. Typing a closer (a `)]}` bracket alone on a line, or a completed keyword `end`/`fi`/`done`/… via `completesCloserKeyword`) re-aligns the line to its opener via `Indenter.closerAlignIndent` (nearest shallower previous line) — done on the completing keystroke, not on Enter, so it never over-dedents. Indent unit (tab vs `tabSize` spaces) is inferred from the file (`detectUnit`). **Smart backspace**: a `BACK_SPACE` `KEY_PRESSED` filter (also in `addAutoIndent`) does a one-press indent delete when the caret is in leading-only whitespace — pure/unit-tested `Indenter.smartBackspaceCount(beforeCaret, afterCaret, hasPreviousLine)`: on a **blank (auto-indented) line** it removes the indent **and the preceding newline** (so one press jumps back to the end of the previous line — "back to where you hit Enter"), on an **indented content line** it clears the indent only, and returns 0 elsewhere. The filter consumes only when the count is >1 so a normal single-char Backspace runs everywhere else. - `git/` — **native-CLI Git integration** (no JGit; we shell out to the user's `git`). `ProcessRunner` is the *only* place Editora runs a subprocess: `ProcessBuilder` + concurrent stdout/stderr drainers + a hard timeout, `LC_ALL=C` and `GIT_OPTIONAL_LOCKS=0` for stable parsing and lock-free status. `GitService` is the facade: every command runs on a single daemon executor (the `highlightExecutor` idiom) and posts results to the FX thread via `Platform.runLater`; `git --version` presence and `rev-parse --show-toplevel` repo roots are cached, and `refresh(contextPath, diffFile, cb)` (status `--porcelain=v2 --branch` + `diff -U0 HEAD -- file`) carries a generation guard so stale results are dropped. The **Commit** tool window is shown only inside a repo via **`ToolWindowManager.setAvailable(tw, inRepo)`** — a *transient* hide (distinct from the persisted `setVisible`; guarded by `shouldShowButton = isVisible && !unavailable`) so it never clobbers the user's show/hide preference; driven from `applyGitState`. `GitPanel` (a `ToolWindowContent` modeled on `BookmarksPanel`) groups Staged/Changes/Untracked with stage/unstage/discard + a commit box (Ctrl/Cmd+Enter), routing mutations through an `Actions` callback. The whole integration is **gated by `Settings.gitSupport`** ("Enable Git", Settings → Git, default **off**): `MainController.applyGitSupport()` (mirrors `applyProjectSupport` — called at init + on every settings apply) disables the status-bar segment (`StatusBar.setGitEnabled`), hides the Commit window (`setAvailable(false)`), clears every open buffer's gutter bars, and the **`ifGit(...)`** wrapper makes each Git command/keybinding a no-op; `git.*`/`tool.commit` are filtered from the command palette + tool-window picker, and `refreshGit()` early-returns when off. `MainController` owns the wiring: it keeps `currentRepoRoot`, runs `refreshGit()` on tab switch / window focus-regain / save / auto-save / project switch (all coalesced by the service's generation guard), and `applyGitState(RepoState)` updates the status bar + `GitPanel` + active buffer's change bars in one pass. After a **git mutation** (commit/stage/unstage/discard/checkout/pull/push/refresh) it instead calls **`afterGitMutation()`**, which does `refreshGit()` *plus* re-diffs every **other open buffer in the same repo** (via `GitService.diff`) so a commit clears the gutter bars on background tabs too, not just the visible one. Git command **failures** are shown via **`gitError(summary, detail)`** — a scrollable error dialog (the often-multiline git stderr is unreadable in the one-line status bar) plus a short status summary. Commands (all no-op with a clear status when git is absent / not a repo): `git.commit` (`C-x g`, opens the Commit tool window + focuses the message), `git.stageFile`, `git.switchBranch` (**toggles** `BranchPopup` — clicking the git status segment opens the dropdown, clicking again closes it; `MainController.chooseBranch` early-returns when `branchPopup.isShown()`/`justHidden()`, a `lastHiddenAt` guard absorbing the autoHide-then-click race like the message-log popup — an IntelliJ-style dropdown anchored above the status-bar branch segment via `StatusBar.gitSegmentNode()`: a search field over actions (New Branch/Pull/Fetch/Push/Commit) + Local/Remote branch sections from `GitService.branches`→`Branches{local:List<BranchInfo>, remote:List<String>}` (one `for-each-ref` with `%(upstream:short)`+`%(upstream:track)`, parsed by the unit-tested `GitService.parseTrack`); the header shows the **remote URL** (`Branches.remoteUrl` — origin's, else the first remote's); each local row shows its **upstream** + **incoming/outgoing** (`↓behind`/`↑ahead`) badge + tooltip ("Git Branch: … / N incoming commits"); local rows → `checkoutBranch`, remote rows → `checkoutRemoteBranch` (`checkout --track`)) and `git.newBranch`, `git.fetch`/`git.pull`/`git.push` (`runNetwork`, longer timeout; **`gitPush()`** uses `--set-upstream origin <branch>` when the branch has no upstream, else plain `push`), `git.refresh`, and `git.clone` (one `Dialog` asks for **URL + destination directory** — a Browse button plus directory auto-fill to `<home>/<repo>` from the URL via the pure/unit-tested `MainController.repoNameFromUrl`; `GitService.clone` runs in the destination's parent; then `openClonedEntry` opens a file from the clone — its README if present — so Git activates **without** creating a project; clone/Projects are independent — also offered via a button in the empty `GitPanel`); the **Commit** tool window (id `commit`, title "Commit") is `tool.commit` (`M-4`) — the name "git" is intentionally left free for future use. Branch switch / pull / push call `reloadAllFromDiskSilently()` to refresh unmodified open buffers whose files changed on disk. **No new dependency / `module-info` change** — it's pure CLI. The diff viewer (below) reuses this facade — `GitService.show(root, spec, cb)` fetches a blob via `git show HEAD:/:/<ref>:rel`, `log(root, file, max, cb)` lists commits (pure `parseLog`), and the pure `repoRelative(root, file)` builds the `git show` spec. **History / Log, blame, and stash** (IntelliJ/VSCode-parity "Core trio", all `ifGit`-gated): `GitService` adds `blame(root, file, cb)` (`git blame --line-porcelain` → the pure, unit-tested `BlameParser` → `BlameLine{hash, author, epochSeconds, summary, uncommitted}`; all-zero sha ⇒ uncommitted), `commitFiles(root, hash, cb)` (`git diff-tree --no-commit-id --name-status -r -M` → the pure `parseNameStatus` → `CommitFile{status, path, origPath}`), `stashList(root, cb)` (`git stash list` → the pure, unit-tested `StashParser` → `StashEntry{index, ref, branch, subject}`), and reuses `run(...)` for the mutations (stash push/pop/apply/drop; `checkout`/`reset --soft|--mixed|--hard`/`revert --no-edit`/`cherry-pick <hash>`). The pure `RelativeTime.of(epoch, now)` buckets an age into `(Unit, value)` for blame labels (unit-tested). **Git Log tool window** (`ui/GitLogPanel`, a `ToolWindowContent` modeled on `GitPanel`; id `gitLog`, `Side.BOTTOM`, `tool.gitLog`/`M-g h`, `Icons::gitLog`, `setAvailable(inRepo)` from `applyGitState`): a commit `ListView` over the selected commit's changed files; selecting a commit fetches its files (`commitFiles`), double-clicking a file opens a **read-only** diff (commit~1 vs commit via `openDiff`), and a commit's context menu offers Copy Hash / Checkout / Soft·Mixed·Hard Reset / Revert / Cherry-Pick / New Branch (each → `MainController.gitMutate` → `gitService.run` → `afterGitMutation` + reload log). Whole-repo log = `tool.gitLog`; the active file's history = `git.fileHistory` (also the tab context menu's "Show File History"); `loadGitLog(file)` caps at 200 commits. **Inline blame** (GitLens-style, current line only): `editor/BlameOverlay` mirrors `InlineValuesOverlay` (a mouse-transparent `Canvas`, coalesced per pulse, 1×1 when off) painting a faint italic `"<author>, <relative-time> • <summary>"` after the caret line; `EditorBuffer.setBlame(List<editor/BlameInfo>)`/`blameHashAtCaret()` keep `editor` git-free (the neutral `BlameInfo{text, hash}` is mapped from git blame by `MainController.toBlameInfos`, which formats the time via i18n + `RelativeTime`). The blame subprocess runs **once per file** (off-thread, re-run only on the coarse `applyGitState` events — tab switch / focus / save / mutation), and caret moves just repaint the cached caret-line annotation — no per-keystroke work. Gated by `Settings.gitBlameInline` (default off; schema 20→21 additive-identity; `MainController.applyGitBlame`/`gitBlameEnabled` = `gitEnabled() && setting`; `git.toggleBlame`/`M-g a`; Settings → Git checkbox + `SettingsWindow.syncGitBlameCheck`). `git.blameShowCommit` opens the caret line's commit diff. **Stash** commands (`git.stash` with an optional message prompt, `git.stashPop`, `git.unstash`/`git.stashDrop` over a `QuickOpen<StashEntry>` picker) plus **Stash/Unstash** rows in the `BranchPopup` actions list. All gated by `gitEnabled()` (so disabled in Simple UI mode); `git.*`/`tool.gitLog` are palette-filtered when Git is off. *Deferred (later phase): a richer log graph/filtering, blame for all lines (the overlay already holds the whole-file map), live-as-you-type gutter diff vs the HEAD blob, interactive rebase/merge/tag UIs.*
- `diff/` — **Diff viewer + merge-conflict resolution** (SDD-driven). Pure, unit-tested core on **java-diff-utils** (`io.github.javadiffutils`, an automatic module — moditect injects a descriptor for the jlink dist, like pdfbox; deps only `java.base`, not signed so no strip). No diff-match-patch: its Maven artifact isn't a reliable JPMS module, so intra-line word highlighting is done in-house. **`DiffEngine.compute(left, right)` → `DiffModels.DiffModel`** walks the java-diff-utils `Patch` deltas into aligned **side-by-side rows** (`Row{type EQUAL/ADDED/REMOVED/MODIFIED, left, leftLine, right, rightLine, leftWordRanges, rightWordRanges}` — a CHANGE delta is paired line-by-line into MODIFIED rows + filler for the longer side) plus derived **unified rows** and `changeBlockStarts` (prev/next nav); `lines(text)` normalizes CRLF + drops one trailing newline. **`InlineDiff`** tokenizes a changed line into word/non-word runs, diffs the token lists, and maps changed tokens back to `[start,end)` char ranges (word-level emphasis, no second lib). **`ConflictParser`** parses Git markers (`<<<<<<<`/`|||||||` base-skipped/`=======`/`>>>>>>>`) → ours/theirs segments + `resolve(file, choices)` (OURS/THEIRS/BOTH/UNRESOLVED-keeps-markers). **`PatchWriter.unifiedDiff`** → a `.patch` via `UnifiedDiffUtils`. **`DiffService`** runs `DiffEngine` off the FX thread (the `GitService` idiom; posts `null` past a 60k-line guard). **UI** (`ui/`): **`DiffViewerPane implements TabContent`** — side-by-side (default) is two read-only `editor-area` `CodeArea`s with synced vertical scroll (guarded `estimatedScrollYProperty` listeners; aligned row counts mean equal line heights), per-side original-line-number gutters (`setParagraphGraphicFactory`), TextMate syntax (`TextMateHighlighter.compute`) **overlaid** with intra-line `.diff-word` spans (`StyleSpans.overlay`), and per-line diff backgrounds via `setParagraphStyle` (`.diff-added/-removed/-modified/-filler`); a toolbar toggles **unified** (one CodeArea, `+/−` gutter), navigates prev/next change, and exports the patch. **`MergeViewerPane implements TabContent`** lists each conflict (ours/theirs `TextArea`s) with Accept Ours/Theirs/Both + a resolved-count toolbar; Save routes the resolved lines back (replaces the active buffer text, undoable). **`Icons.diff()`/`merge()`** + `.diff-*`/`.merge-*` CSS in `app.css`. **MainController** entry points: `diff.vsHead` (`C-x v =`; tab menu "Compare with HEAD"; Git-panel "Show Diff" → `diffGitPanelFile(rel, staged)` = staged: index↔HEAD, unstaged: working↔index), `diff.compareWith` (FileFinder picks the second file — **ungated**), `diff.vsCommit` (`QuickOpen` over `GitService.log`), `merge.resolve` (opens the merge view when `ConflictParser.hasConflictMarkers`). Git-backed diffs are `ifGit`-gated; the working side uses the open buffer's live text when the file is open (canonical-path match), else disk. **Live refresh**: each side is a re-fetchable `DiffSide` (a `Consumer<Consumer<String>>` — a `git show` blob or `worktreeText`), and `openDiff` installs a `DiffViewerPane.refresher` that re-fetches both + `pane.updateContent(...)` **only when the content changed** (`pane.matches(l,r)` guard, so an unchanged refresh keeps the view + scroll); `refreshOpenDiffs()` (iterates tabs whose `userData instanceof DiffViewerPane`) runs on **window focus-regain** (a compared file changed on disk) and in **`afterGitMutation`** (a commit/stage/checkout moved HEAD/index/working). **Apply change** (IntelliJ's hunk-accept arrow): `DiffViewerPane.setEditable(EditableSide, onApply)` marks which side is the writable local file (RIGHT for vs-HEAD/vs-commit/unstaged, LEFT for compare-with, NONE for staged index↔HEAD); that side's gutter then shows apply chevrons (`.diff-apply`): a **single chevron on every changed row** (per-line / sub-hunk — `applyRow` → `computeApplied(i,i+1)`) and a **double chevron at each change block's first row** (whole hunk — `applyBlock(start)` → `computeApplied(start, blockEndFrom(start))`), both over the pure aligned rows; an **Apply-all** toolbar button (`Icons.check()`, shown only when a side is editable) replaces the editable file with the other side wholesale. Both hand the new full text to `MainController.applyToLocal(target, text)` — which routes through an **undoable editor buffer** (`bufferForApply`: the open buffer, else one opened in the **background** via `addBuffer(buffer, false)` so the diff stays focused), `replaceText`s it (marks dirty), then `refreshOpenDiffs()` re-diffs. The toolbar's **Undo** (`undoLocal` → the buffer's `undo()`) and **Save** (`saveLocal` → `save(buffer)`) act on that buffer, so applies are revertible and persisted explicitly. **Apply-all** confirms first (it replaces the whole file). (`setEditable` rebuilds the view so the arrows + Undo/Save/Apply-all toolbar buttons appear, since it runs after construction.) **One new dependency + its moditect entry; no `module-info` opens / Settings / schema change.** *Deferred: 3-way merge with base column, diff-against-arbitrary-ref browsing, in-diff editing, gutter-arrow stage/revert hunks, the SDD's standalone Diff Model persistence.*
- `process/` + `mermaid/` — **Mermaid diagram support** (off by default, gated by `Settings.mermaidSupport`). The generic subprocess runner `ProcessRunner` lives in **`com.editora.process`** (moved out of `git/` so Git and Mermaid share it; `GitService` passes `GIT_OPTIONAL_LOCKS=0` through the new `run(dir,timeout,cmd,extraEnv)` overload, and `LC_ALL=C` is always set). **GUI-launch PATH fix (important):** a Finder-launched `.app` (or a `.desktop`) inherits a stripped `PATH` (`/usr/bin:/bin:…`) without Homebrew/npm/Node dirs, so `mmdc`/`npx`/`maid` (and a Homebrew `git`) wouldn't be found — `ProcessRunner` builds an **augmented PATH** = the inherited PATH **+ the user's login-shell PATH + the usual install dirs**, then rewrites a bare command name to its absolute path against it before launching (Java's Unix `ProcessBuilder` resolves the executable against the JVM's own stripped PATH, not the child env, so both steps are needed). The **login-shell PATH** (`ProcessRunner.loginShellPath()`) is the crucial part for version managers: it runs the user's `$SHELL -l -i -c 'printf …$PATH…'` **once** (stdin `/dev/null`, stderr discarded, 5 s timeout, markers fence the PATH off from any rc banner — the pure marker parse `extractMarked` is unit-tested; the VS Code `resolveShellEnv` approach) and folds its entries in. This recovers bin dirs that are **version-specific and so can't be hardcoded** — e.g. **nvm**'s `~/.nvm/versions/node/<ver>/bin` (where npm-global servers like `typescript-language-server`/`pyright-langserver`/`yaml-language-server` live), fnm/asdf/volta, and a profile-set Homebrew. The hardcoded `EXTRA_PATH_DIRS` (`/opt/homebrew/bin`, `/usr/local/bin`, `/opt/local/bin`, `~/.local/bin`, `~/go/bin`, `~/.cargo/bin`, …) remain as a fallback. `augmentedPath()` is cached after the first call, and `LspManager`'s constructor **warms it off-thread** (the probe spawns a process) so neither detection nor the FX-thread `session.start()` blocks. Windows yields no login-shell PATH (no POSIX convention). A Settings absolute path still overrides. `com.editora.mermaid.Mermaid` is the static CLI facade over **mmdc** (mermaid-cli; renders/exports) and **maid** (`probelabs/maid`; a fast pure-JS linter): `renderPng(mmdc,source,dark)` (writes a temp `.mmd`, runs `mmdc -i -o -t default|dark -b transparent -s 2`, reads the **PNG** — *not* SVG, because JavaFX/JSVG can't apply mermaid's embedded `<style>` CSS or render its `<foreignObject>` HTML labels, so SVG comes out all-black/textless; mmdc rasterizes faithfully via headless Chromium), `exportTo(...)` (format by the chosen file extension — SVG/PNG/PDF, mmdc writes it directly), `validate(maid,source)` (runs `maid --format json`, parsed by the pure, unit-tested `MaidOutput` → `Diagnostic{line,column,severity,code,message}`; JSON with a text fallback). `MermaidService` (UI-owned, mirrors `GitService`: daemon executor + `Platform.runLater` + cached availability probe) backs the export command + the Settings detection status. Because `MarkdownRenderer` is synchronous/pure but mmdc is a multi-second subprocess, the editor-side **`MermaidImages`** static façade (like `PreviewImageLoader`: daemon pool + cache, configured via `configure(enabled,mmdcExe,maidExe,dark)` injected from `MainController.applyMermaidSupport`) returns a placeholder `StackPane` immediately and fills it async — decoding mmdc's PNG with a JavaFX `Image` (logical width = pixels ÷ `RENDER_SCALE`) into an `ImageView` sized by a caller-supplied `DoubleUnaryOperator` (embedded Markdown blocks clamp to the content column; a standalone `.mmd` multiplies by the preview zoom so zoom actually resizes the image) — **cached by `sha-256(source+theme)`** so the debounced whole-document re-render never re-invokes mmdc for unchanged diagrams; on failure it shows `maid`'s precise line/column diagnostics (else mmdc's stderr) in a `.mermaid-error` label. (No JSVG/AWT in the Mermaid path — PNG decodes natively.) `MarkdownRenderer` special-cases a fenced block whose info string is `mermaid` (`isMermaidInfo`) → `MermaidImages.node(...)`, else the usual `codeBlock`. Standalone **`.mmd` files** (TextMate grammar `grammars/mermaid.tmLanguage.json` → `source.mermaid`, `LanguageRegistry` `mmd`→`mermaid`) get the same 3-mode preview: `EditorBuffer.isDiagram()`/`hasPreview()` generalize the markdown gating, and `scheduleRenderPreview` branches to render the whole buffer as one diagram. Command `mermaid.export` (`FileChooser` SVG/PNG/PDF → `MermaidService.export`). Each tool is run as a **tokenized command** (`Mermaid.command(configured, default)`, whitespace-split) so a bare binary *or* an npx invocation works: defaults are `mmdc` (a binary; the user installs mermaid-cli) and **`npx -y @probelabs/maid`** (maid is normally run via npx, not a global bin — `MermaidService.DEFAULT_MMDC`/`DEFAULT_MAID`). maid emits its JSON report on stdout regardless of exit code, so `validate` always parses it (doesn't gate on success); the real shape is `{errors:[{line,column,message,severity,code,hint}], …}` — locked in by a `MaidOutputTest` case. Settings → **Mermaid** page: enable checkbox + `mmdc`/`maid` command fields (Browse; blank = the defaults above) + a live found/not-found status (`detect` counts any clean launch, not just exit 0). `MainController.applyMermaidSupport()` (mirrors `applyGitSupport`, init + every settings/theme apply) configures the façade + service + re-renders open previews; `ifMermaid(...)` no-ops the command when off and `mermaid.*` is filtered from the palette. **No new Maven dependency / `module-info` change** (grammar rides the existing `opens com.editora.grammars`; mmdc/maid are external CLIs the user installs via npm). **Live linting**: `.mmd` buffers run **maid** debounced (~450 ms) while typing via `MermaidService.validate(source, cb)` (async, gen-guarded) injected into `EditorBuffer.setMermaidValidator`; `MermaidLintOverlay` (a `Canvas` mirroring `SpellCheckOverlay`) draws red wavy underlines at each diagnostic's `line/column/length` (visible paragraphs only, coalesced), and an `area` `MOUSE_MOVED` handler shows the maid message in a tooltip over a squiggle. Gated by `EditorBuffer.setMermaidLintEnabled` (controller sets it = mermaid on && maid detected, re-applied after the async `detect`). **Autocomplete**: `.mmd` buffers get Mermaid keyword completions (`completion/MermaidKeywords`, merged in `CompletionEngine.complete` when `snippetLang == "mermaid"`) + bundled `snippets/mermaid.json`, gated by `Settings.autocompleteMermaid` (Settings → Editor, default on; `view.toggleAutocompleteMermaid`) — **effective only** when the master autocomplete is on, Mermaid is enabled, and mmdc is detected (`MainController.effectiveMermaidAutocomplete()`; `EditorBuffer.updateCompletion` routes diagram buffers through this flag). `MaidOutput.Diagnostic` carries `length` for the underline span. *Deferred (per the SDD): a Markdown→HTML/PDF export pipeline (none exists to hook "render before export" into), a Mermaid LSP (no LSP layer exists), and block-at-caret export from Markdown.*
- `lsp/` — **Language Server Protocol** support (off by default, gated by `Settings.lspSupport`). **Servers: Java** (Eclipse **JDT LS**, `jdtls`), **TypeScript/JavaScript** (`typescript-language-server --stdio`, serving js/ts/jsx/tsx), **Python** (`pyright-langserver --stdio`), **XML** (`lemminx`, serving the `xml` language id — xml/xsd/xsl/svg/fxml/pom/…), **JSON** (`vscode-json-language-server --stdio`), **Bash/Shell** (`bash-language-server start`, serving the `shell` language id — sh/bash/zsh; its diagnostics come from `shellcheck` when that's on PATH), **YAML** (`yaml-language-server --stdio`), **Go** (`gopls`, root markers `go.mod`/`go.work`), **Rust** (`rust-analyzer`, root marker `Cargo.toml`), **PHP** (`phpactor language-server`, root marker `composer.json`), **Ruby** (`ruby-lsp`, root markers `Gemfile`/`.ruby-version`), **C/C++** (`clangd` — one server serving both the `c` and `cpp` language ids, root markers `compile_commands.json`/`CMakeLists.txt`/…), **HTML** (`vscode-html-language-server --stdio`), **CSS** (`vscode-css-language-server --stdio`), **Kotlin** (`kotlin-language-server`), **Lua** (`lua-language-server`), **Dockerfile** (`docker-langserver --stdio`), **SQL** (`sqls`), **Terraform** (`terraform-ls serve`, root markers `.terraform`/lock file), **TOML** (`taplo lsp stdio`), and **C#** (`csharp-ls`, root markers `global.json`/`.git`) — **twenty-one servers** total. Most already had highlighting + folding, so each was just a registry/Settings entry; the five with **no grammar** (PHP, Lua, Dockerfile, Terraform, TOML) each got a bundled `*.tmLanguage.json` (PHP `source.php` from VS Code MIT; Lua/Dockerfile/Terraform/TOML `source.lua`/`source.dockerfile`/`source.hcl.terraform`/`source.toml` from the shiki tm-grammars collection — see NOTICE) + `LanguageRegistry`/`FoldRegions`/`Indenter`/`Commenter` entries, like the JS/TS case. **Dockerfile** is matched by **filename** (extension-less `Dockerfile`/`Containerfile`, or a `Dockerfile.<tag>` suffix) via a special case in both `LanguageRegistry.forFileName` and `GrammarRegistry.scopeForFileName`; TOML moved off the borrowed INI grammar onto its own `source.toml`. **Adding a server = one `ServerDef` in `LspServerRegistry` + its id in `MainController.LSP_SERVER_IDS` + the served language id(s) in `EditorBuffer.LSP_LANGUAGES` + a Settings command/enable pair** (the Settings → LSP page is **data-driven** over a `SettingsWindow.lspServerUis()` descriptor list, so a new server is one entry there too) — `LspManager.configure(enabled, Map<serverId,command>)` is map-based so it never needs a signature change. The registry is **server-centric**: `LspServerRegistry` lists each server with the language ids it serves + its root markers (Java: `pom.xml`/`build.gradle`/…; TS: `tsconfig.json`/`jsconfig.json`/`package.json`/`.git`), so `LspManager` keys **one session per `(serverId, root)`** and js/ts/jsx/tsx in one project share a single `tsserver`. Each server has its own configurable command (`Settings.javaLspCommand`/`typescriptLspCommand`; blank ⇒ default) + its own cached availability probe; `MainController` detects each independently and gates each buffer by *its* server's availability and the file's language id (`EditorBuffer.getLanguage()`, which here doubles as the LSP languageId: `java`/`javascript`/`javascriptreact`/`typescript`/`typescriptreact`/`python`/`xml`/`json`/`shell`/`yaml`/`go`/`rust`/`php`/`ruby`/`c`/`cpp`/`html`/`css`/`kotlin`/`lua`/`dockerfile`/`sql`/`terraform`/`toml`/`csharp`). **JS/TS also get syntax highlighting** — bundled `typescript.tmLanguage.json` (`source.ts`) + `typescriptreact.tmLanguage.json` (`source.tsx`) grammars from VS Code (MIT, see NOTICE); `.ts/.mts/.cts/.js/.mjs/.cjs`→`source.ts`, `.tsx/.jsx`→`source.tsx` (the TS grammar tokenizes plain JS, so no separate JS grammar), with the extensions registered in `LanguageRegistry`/`FoldRegions`/`Indenter`/`Commenter`. **Auto-imports**: accepting an LSP completion runs an optional `Completion.onAccept` hook (`CompletionMapper.map(items, onAcceptFor)`); for an item that `mayHaveAdditionalEdits` (carries resolve `data`), `MainController.autoImportAccept` → `LspManager.resolveCompletion` (`completionItem/resolve`) and applies the resolved `additionalTextEdits` (e.g. a TS `import` line) via `EditorBuffer.applyLspEdits(List<editor/LspTextEdit>)` (offset-mapped, applied bottom-to-top; the `LspTextEdit` record keeps `editor` lsp4j-free). Uses **LSP4J** (`org.eclipse.lsp4j` + `.jsonrpc`, automatic modules → moditect-patched in the `dist` profile like tm4e/pdfbox; jsonrpc reuses the existing `gson`). `module-info` **`opens com.editora.lsp;`** (unqualified, **not** qualified-to-jsonrpc) because Gson reflectively reads our `@JsonNotification("language/status")` param DTO (`LanguageServerSession.LanguageStatus`) and, under `mvn javafx:run`, runs in the **unnamed** module — a qualified opens to `org.eclipse.lsp4j.jsonrpc` alone leaves it unable to set the DTO fields accessible. **`LanguageServerSession`** runs one external server process per workspace root over stdio: it launches via `ProcessBuilder` reusing `ProcessRunner.resolveExecutable`/`applyStandardEnv` (the GUI-launch PATH augmentation, so a Finder-launched `.app` finds `jdtls`), drives it with an LSP4J `Launcher` on a daemon executor, implements `LanguageClient` (receives `publishDiagnostics`/log/show-message), does `initialize`/`initialized` + client capabilities, full-text document sync (`didOpen`/`didChange`/`didSave`/`didClose`), the request subset (completion/hover/definition/references), **queues requests until initialize completes**, and `shutdown`+`exit` on dispose. **The server's stderr is `Redirect.DISCARD`ed** — nothing drains it, so leaving it a PIPE lets a chatty server (jdtls logs heavily) fill the ~64 KB OS pipe buffer and block mid-startup (LSP traffic is on stdout). **Dispose kills the whole descendant process tree** (`process.descendants()` snapshot → `destroy`, then the root), not just the launcher: jdtls (and others) is a wrapper script (Homebrew `jdtls` → python → java), so destroying only the wrapper **orphans the real server JVM**, which keeps running and holds its Eclipse workspace `.lock` — the next session for the same root then can't acquire the lock and hangs (no diagnostics/completion). - `dap/` — **Multi-language debugging (Debug Adapter Protocol)** for **Java**, **Python**, and **JavaScript (Node)**, off by default (gated by `Settings.debugSupport`). ## Conventions

- **Performance is a top priority.** Editora must stay fast and responsive, especially on large files. Treat the UI thread as sacred and these as the hot paths: typing/editing, scrolling, syntax highlighting, the document overlays (whitespace, minimap, 80-column ruler), and the line-number gutter. Rules to uphold:
  - **Never block the JavaFX Application Thread** with non-trivial work — tokenize/parse/search off-thread (see the `highlightExecutor` pattern in `EditorBuffer`) and apply results back on the FX thread with a generation/stale-result guard.
  - **Debounce and coalesce** — re-highlighting is debounced; overlay/ruler/minimap redraws coalesce to one per pulse (a `pending` flag + `Platform.runLater`). Don't add per-keystroke or per-scroll-pulse work that isn't coalesced.
  - **Work incrementally / only on what's visible** — highlighting re-tokenizes only from the changed line; overlays iterate just the visible paragraphs (`firstVisibleParToAllParIndex … last`) and skip folded lines. Avoid O(document) work on edits or scrolls.
  - **Don't defeat the per-node CSS style cache** — keep token rules as the compound `.text.<class>` selector (see `syntax.css`); coalesce adjacent same-style spans (`SpanMerger`) before `setStyleSpans`; never query `getCharacterBoundsOnScreen` synchronously inside a layout/viewport event.
  - **Large/huge files** already disable highlighting + minimap (≥5 MB) and go read-only with a capped load (≥50 MB); preserve those guards. Bound memory (undo history is capped; loads are capped).
  - **Bound retained GPU textures.** JavaFX's Prism texture pool has a fixed ceiling (default 512 MB); exhausting it makes the render thread NPE (null `RTTexture`/mask texture) → a black window — seen only in the packaged build. So don't let GPU-backed resources grow with the number of open files: a background (non-selected) tab's buffer drops its minimap snapshot via `EditorBuffer.setRenderingActive(false)` (driven from the tab-selection + tabs-added listeners; `Minimap.setRenderingActive` nulls the cached snapshot and stops rendering, regenerating on show), and the `PreviewImageLoader`/`MermaidImages` image caches are **LRU-bounded** (not unbounded `ConcurrentHashMap`s) since each holds an `Image` (a pinned texture). The `dist` jpackage app (and `mvn javafx:run`) also raise the caps as a safety net (`-Dprism.maxvram=2G -Dprism.maxTextureSize=16384`). When adding any per-buffer Canvas/Image, make sure it's released or bounded.
  - **Packaged-runtime JVM tuning (footprint + install size).** The `dist` jpackage `<javaOptions>` (mirrored into the `javafx:run` options so dev == prod) cap the heap and GC: **`-Xmx2g`** (without it, jpackage inherits the default ergonomic max heap ≈ ¼ of physical RAM — multi-GB on a high-RAM box — which lets GC-driven RSS balloon; 2g is predictable across the release matrix and safe for a 50 MB huge-file load + capped undo) and **`-XX:+UseSerialGC`** (a mostly-idle single-user editor on a small heap gets sub-ms pauses with no parallel GC-thread overhead — lower idle RSS than the default G1; if large-file GC *hitching* ever shows up, generational ZGC is the smoothness-over-footprint alternative). The jpackage `<jLinkOptions>` strip the bundled runtime (`--strip-debug --no-man-pages --no-header-files --compress=zip-6`) → smaller DMG/MSI/DEB + less mapped footprint. (`--strip-native-commands` is deliberately *omitted* from jLinkOptions — `bin/java` must survive the jlink step for the AOT training run below; the `aot_build` helper deletes `bin/` itself afterward, reclaiming the same footprint. The runtime's `bin/` is never used at launch — the jpackage native launcher starts the JVM via `libjli`, and every external tool (java/python/node/git/LSP/DAP) is resolved via the augmented PATH.) **AOT cache (JDK 25 Leyden) — implemented, ~28%/≈300–480 ms faster cold start** (measured macOS: first-frame ~1060 ms → ~760 ms; delivered-app wall delta ~480 ms): the build does a **two-phase jpackage** — phase 1 (`prepare`) jlinks an `APP_IMAGE` into `target/aot-image` with `-XX:AOTCache=$APPDIR/editora.aot` baked into the launcher `.cfg`; then `scripts/aot_build.java` (run by `exec-maven-plugin` with the build JDK) **trains a full-GUI cache against the image's own runtime** (so archived classes match the shipped runtime, path-independently — it maps after the install-dir move because the classes live in the jlinked `lib/modules`, keyed to the runtime image not absolute paths), writes `editora.aot` into `$APPDIR`, strips `bin/`, then either copies the image (APP_IMAGE builds) or wraps it into the installer via `jpackage --app-image` (DMG/MSI/DEB). **A *headless* trainer gives ≈0 win** — the gain is JavaFX scene/control/CSS class loading, which only loads when a window renders; the training run is the real GUI driven by `-Deditora.aotTrainExit` (`WindowManager` renders the first frame, settles ~2.5 s for highlight-class coverage, then `System.exit`). On Linux the helper auto-wraps training in `xvfb-run` (release.yml installs `xvfb`+GTK/GL libs on the Linux legs); macOS/Windows runners use their session. **Fully failure-tolerant:** if training fails (no usable display on a runner, etc.) the build continues and ships without the cache — a missing/invalid `-XX:AOTCache` file logs to stderr (invisible in the packaged app) and starts normally under the default `AOTMode=auto`. **Cost: the cache is ~60 MB**, so the installed image grows accordingly (the DMG/MSI/DEB compress it — the DMG download stayed ~81 MB). **Commented-out opt-ins in the pom (DEVICE-TEST per platform before enabling, they touch correctness):** the macOS **Metal** render pipeline (`-Dprism.order=mtl,es2,sw`, macOS-only — Apple deprecated the es2/OpenGL backend) and JDK 25 **compact object headers** (`-XX:+UseCompactObjectHeaders`, experimental, ~10–20% live-heap cut — note it changes the AOT cache's recorded layout, so retrain after enabling). *(Aside: `App.main` exits `--version`/`--help` via `System.exit(0)`, not a bare return — launching this `Application` subclass in module mode pre-starts the FX toolkit, whose non-daemon keep-alive thread would otherwise hang the process after `main` returns.)*
  - **Assess and report the performance cost of every change.** For any implementation or bug fix, evaluate its cost on the hot paths (allocation per keystroke/scroll, added FX-thread work, extra layout/CSS passes, memory) and tell the user about it — even if it's "negligible." If a change risks a regression, measure (e.g. the temporary `System.nanoTime` instrumentation used for the startup investigation) rather than guess.
- **Code style — Palantir Java Format, enforced by Spotless.** All Java is auto-formatted with [Palantir Java Format](https://github.com/palantir/palantir-java-format) (a lambda-friendly, 120-col fork of google-java-format) via the `spotless-maven-plugin` (config in `pom.xml`; versions in the `${spotless.version}`/`${palantir.version}` properties). **Run `mvn spotless:apply` before committing** — `mvn spotless:check` is bound to the `verify` phase, so `mvn verify`/`mvn package`/CI fail on unformatted code (the `mvn javafx:run`/`mvn compile` dev loop is deliberately left untouched). Pipeline: `removeUnusedImports` → `palantirJavaFormat` → `importOrder` → `toggleOffOn`. **Import groups** (blank-line separated; longest-prefix wins, so `javafx` is *not* swallowed by `java`): JDK (`java`/`javax`) → `javafx` → everything else (third-party + `com.editora`) → static imports last. Escape hatch for hand-aligned code: wrap it in `// spotless:off` … `// spotless:on`. Palantir 2.72.0+ supports JDK 25. The two tree-wide mechanical reformats are recorded in `.git-blame-ignore-revs` (enable locally once with `git config blame.ignoreRevsFile .git-blame-ignore-revs`; GitHub honors it automatically) so `git blame` skips them.
- **Adding a feature = add a `Command`, and surface it in the command palette.** Every time we add a command or an applicable user-facing feature, it must be discoverable in the palette. Register it in `MainController.registerCommands()` and, if it needs a key, add a binding to `emacs.json`. The palette is populated from `CommandRegistry.all()`, so a properly registered command appears automatically — never add a user-facing action that bypasses the registry. Toolbar buttons and the palette both dispatch through commands; don't wire UI handlers to logic directly when a command fits.
- **Every user-facing string is localized — never hardcode UI text.** Any new label, status/echo message, tooltip, dialog title/header/body, button, menu item, prompt, or placeholder must go through the i18n catalog: `import static com.editora.i18n.Messages.tr;` then `tr("some.key")` (or `tr("key", args…)` for `MessageFormat` parameters like `Saved {0}`) — never pass a raw English literal to a JavaFX control. **Add the key to all six `resources/com/editora/i18n/messages[_<lang>].properties` files** (English base + `it/es/fr/pt/de`) with a real translation in each; `MessagesTest`'s key-parity check fails the build if any locale is missing a key. New `Command`s get their title for free via `Command.of(id, runnable)` + a `command.<id>` key. Only deliberate exceptions stay literal: technical tokens (`UTF-8`, `LF`/`CRLF`, `Ln`/`Col`, git verbs in `gitlabel.*`, detected language names, example URLs) and the pre-GUI `--help`/`--version` text. See the `i18n/` architecture bullet for the full system.
- **App icon / branding:** the source logo is `branding/editora-icon.svg`. Window-icon PNGs (16–512) live in `resources/com/editora/icons/` (loaded into `stage.getIcons()` in `App`; also the About-dialog logo). Native-installer icons `branding/editora.{icns,ico,png}` are generated from the SVG (rsvg-convert + iconutil + a small ICO writer) and passed to jpackage via `${jpackage.icon}`, set per-OS in the OS-activated profiles. Regenerate after editing the SVG.
- **Context-menu icons:** **every right-click `MenuItem` carries a leading icon** (`MenuItem.setGraphic(...)`). UI-package menus (tab strip + Welcome close in `MainController.installTabMenu`, `ProjectPanel`, `GitPanel`, `BookmarksPanel`, `NotesPanel`, `MessageLogPopup`, `FileBreadcrumb`) use `com.editora.ui.Icons` directly; the **editor** package's menus (`EditorBuffer.installContextMenu` + `showPreviewContextMenu`) can't depend on `ui`, so they use **`com.editora.editor.MenuIcons`** — a package-local mirror of `Icons` (same Material 24dp single-path glyphs, `toolbar-icon` style class, 0.8 scale) holding only the glyphs those two menus need (cut/copy/paste/undo/redo/selectAll, markdown bold/italic/strike/code/link, LSP goto-def/refs/hover, spellcheck/add/block, note/run/download/print); the Run item reuses `FoldManager.runGlyph()`. Icons are colored via the global `.toolbar-icon` rule (`-color-fg-muted`, theme-aware), so they track light/dark for free — **no new CSS**. When adding a context menu or item, give every item a `setGraphic` glyph (reuse an existing one where no exact match exists, e.g. the close-tab X across all close-variants). The recent-files toolbar **MenuButton** dropdown is not a right-click menu and is intentionally excluded (its entries already carry an inline remove-icon); the snippet **choice** dropdown lists content values, not commands, and is likewise excluded.
- **Bundled fonts:** five monospace families ship under `resources/com/editora/fonts/<family>/` (Regular/Bold/Italic/BoldItalic TTFs; Fira Code has no italic), **plus Inter** (sans-serif, `fonts/inter/`) as the Markdown **preview + PDF prose** font. `Fonts.load()` registers them all via `Font.loadFont` at the very start of `App.start` (before any CSS), so they resolve by family name regardless of the OS. `Fonts.BUNDLED` (JetBrains Mono first = default) is the **monospace** editor-font picker list and deliberately **excludes Inter** (Inter isn't an editor font); it's listed ahead of the system monospaced families in the Settings font picker. Inter is the preview prose font on **all platforms** (`.markdown-preview .md-text` in `app.css` — pinned explicitly because raw `Text` nodes don't reliably inherit the scene font, and because the macOS chrome's logical "System" family can't be bold-rasterized cleanly by JavaFX) and the PDF prose font (`MarkdownPdfWriter`). They're plain resources (no `module-info`/moditect change) and are picked up by the `dist` build automatically. Attributed in `NOTICE` (all OFL-1.1).
- **Theming:** AtlantaFX is the user-agent stylesheet and themes standard controls (`Themes`, set via `Application.setUserAgentStylesheet`). It does **not** style the RichTextFX `CodeArea` — editor surface, line-number gutter, and syntax token colors live in `styles/app.css` + `styles/syntax.css` (the **Primer Light** defaults). **Editor color themes** (`EditorThemes`) layer an override stylesheet from `styles/editor-themes/<name>.css` onto the scene *after* those defaults (Primer Light = no override); `MainController.applyEditorTheme` swaps that sheet on the scene and sets each buffer's current-line highlight color (RichTextFX's line-highlight fill is set in code, not CSS). The editor theme follows the AtlantaFX theme (`EditorThemes.defaultFor`) until the user picks one (`Settings.editorThemeUserSet`). Token rules must stay the compound `.text.<class>` form — see the header comment in `syntax.css`. The **Project tool window file tree** is themed via two looked-up CSS colors `-project-folder-color`/`-project-file-color` (defined on `.project-tree` in `app.css`, defaulting to AtlantaFX accent/muted); each editor theme overrides them, so a new editor theme should add a `.project-tree { -project-folder-color: …; -project-file-color: …; }` rule.
- **Tests** cover pure logic only (keymap resolution, command registry, config merge, highlighting spans). There is no GUI/toolkit test harness; verify interactive behavior by running the app.

## Packaging note (important)

RichTextFX is a **personal fork** (`io.github.adriandeleon:richtextfx`, vendored in the in-project
`m2-repo/`, source at github.com/adriandeleon/RichTextFX) that adds VS Code–style **multiple cursors +
Alt+drag column/box selection** via a self-contained `org.fxmisc.richtext.multi.MultiCaretController.install(area)`
add-on (a layered wellbehaved `InputMap`, gated on `MultiCaretManager.hasExtras()` so it's transparent with
one caret). **`EditorBuffer`** installs it on `area`/`area2` via `setMultiCaretEnabled(...)` (gated by
`Settings.multiCaret`, default on; pushed by `MainController.applyMultiCaret`), exposes
`hasMultipleCarets()` + delegating `addCaretNextOccurrence`/`addCaretAbove`/`addCaretBelow`/`collapseCarets`
(palette commands `edit.addCaret*`/`edit.collapseCarets`, `view.toggleMultiCaret` — **no keymap bindings**:
gestures are mouse + Esc, native to the fork). Because Editora's area-level KEY filters (auto-indent/close,
snippets, completion, view paging) are capture-phase *filters* that run before the fork's node InputMap,
each one early-returns (no consume) via `multiCaretActiveOn(a)` when that area has extra carets, so editing
fans out to all carets. *Known limitation:* Emacs movement chords (`C-f`/`C-n`/…) are resolved by the
scene-level `KeyDispatcher` on the primary caret and don't fan out — use arrows for multi-caret movement.
To update the fork: rebuild it, copy the new `richtextfx-<version>.{jar,pom}` into `m2-repo/`, bump
`<richtextfx.version>`. RichTextFX and its transitive deps (`reactfx`, `flowless`, `undofx`, `wellbehavedfx`)
are **automatic modules**, which `jlink` cannot link. The `moditect-maven-plugin` in
the `dist` profile injects explicit `module-info` descriptors into them. If you bump
RichTextFX or add a dep it uses, you may need to adjust those descriptors' `requires`
(e.g. several of them need `javafx.controls` for `IndexRange`). Use RichTextFX
**0.11.7+** — earlier versions are incompatible with JavaFX 25.

tm4e (the syntax engine) ships as the NetBeans repackaging
`org.netbeans.external:org.eclipse.tm4e.core-0.14.0:RELEASE260` (tm4e is not on
Maven Central). Its Oniguruma backend (`org.jruby.joni:joni`, `org.jruby.jcodings:jcodings`)
and `com.google.code.gson:gson` are already proper modules, but tm4e core is an
automatic module, so `moditect` injects a `module-info` for it too. The NetBeans
jar is also **code-signed**, and `jlink` rejects signed modular jars — the `dist`
profile's antrun step strips `META-INF/*.SF,*.RSA,*.DSA,*.EC` before linking.
